### 3.2 코드베이스 사실 조사 (플랜 anchor)

#### 3.2.1 어댑터 Protocol 패턴 (`api/app/adapters/`)
- `llm.py` — `LLMProvider(Protocol)` 정의, `def complete(...) -> str: ...` 하나만.
- `factory.py` — `get_llm_provider(purpose)` 로 dispatch, `_resolve_provider()` 는 ENV → provider 문자열 정규화, lazy import 로 SDK 로딩 회피.
- **W5-6 적용 방침**: 동일 5-Protocol 패턴 계승. `PaymentProvider(Protocol)` + `KakaoPayImpl` + `get_payment_provider()` 팩토리. ENV `JETRAG_PAYMENT_PROVIDER` (default `kakaopay`).

#### 3.2.2 022 스키마 (재사용)
- `plans(code PK, name, max_documents INT, answers_per_day INT, price_krw INT DEFAULT 0)`
  - seed: `('free',10,5,0)`, `('pro',200,50,6900)`.
- `subscriptions(user_id UUID PK, plan_code REF plans, status CHECK(active/past_due/canceled) DEFAULT 'active', current_period_end TIMESTAMPTZ, billing_key TEXT, created_at, updated_at)`
- RLS: plans SELECT all (auth+anon), subscriptions SELECT own (auth), service_role full.
- **W5-6 신규 필요**: `billing_key` 는 존재 → **암호화 저장** (Fernet, `cryptography` 라이브러리 — `pyjwt[crypto]` 로 이미 설치됨). 추가로 결제 이력 테이블 `payment_history` (선택 — 실패 로그·재시도 tracking).

#### 3.2.3 quota 서비스 (`api/app/services/quota.py`)
- `_EFFECTIVE_STATUSES = ("active", "past_due")` — `past_due` 는 grace 로 이미 Pro 취급.
- `get_effective_plan(user_id) -> PlanLimits | None` — fail-open.
- **W5-6 결정**: past_due grace 기간 = 결제 실패로부터 **7일**. 8일차 배치가 `canceled` 로 전환 (데이터 보존 — Free 강등).

#### 3.2.4 config ENV 패턴 (`api/app/config.py`)

### 3.3 KakaoPay open-api 사양 (외부, 결제 사양서 anchor)
- Base URL: `https://open-api.kakaopay.com`
- Auth header: `Authorization: SECRET_KEY {secret}`
- 정기결제 flow:
  - `POST /online/v1/payment/ready` — 결제창 URL + `tid` 반환.
  - `POST /online/v1/payment/approve` — 사용자 결제 완료 후 `pg_token` 으로 승인. 정기결제면 응답의 **`sid`** (구독 키) 저장.
  - `POST /online/v1/payment/subscription` — SID 로 정기 결제 실행 (매월 배치).
  - `POST /online/v1/payment/manage/subscription/inactive` — SID 비활성화 (해지).
  - `POST /online/v1/payment/manage/subscription/status` — SID 상태 조회.
- 개발용 sandbox CID: `TCSUBSCRIP` (정기결제 테스트) — production CID 는 심사 후 발급.
