## 5. 검색 파이프라인 최종 흐름

```
사용자 질문
    ↓
[Step 1] NFC 정규화
    ↓
[Step 2] 메타 필터 추출 (날짜·#태그·doc명, regex 룰 기반, 무료)
    │
    ├── 메타만으로 충분 (예: "#법률 자료 목록", "지난주 받은 자료")
    │       → RAG skip → 메타 필터 결과만 표시 (~0.3s)
    │
    └── 일반 query → 다음 단계
    ↓
[Step 3] cheap router intent 분류 (룰 + lightweight score, 무료)
    │
    ├── simple intent (단편·exact_fact)
    │       → 단일 query 검색
    │
    ├── complex intent (cross-doc, 비교, 인과, 변경점, 긴 query, 복수 대상)
    │       → [Step 3a] paid query decomposition (Flash-Lite, ~$0.0002)
    │             sub-query 2~5개 생성
    │       → 각 sub-query 별 검색
    │
    └── confidence < 0.75 (애매)
            → 보수적 분기: paid decomposition 자동 trigger
            → 또는 사용자에게 "분해 검색 시도?" 확인
    ↓
[Step 4] query expansion (사전 30 entry, 무료, default on)
    ↓
[Step 5] Hybrid 검색 (BGE-M3 dense + PGroonga sparse, 무료)
    ↓
[Step 6] doc-level RRF 가산 (doc_embedding cosine, 무료, default on)
    ↓
[Step 7] RRF merge → top 50 candidate
    ↓
[Step 8] BGE reranker (HF 무료, candidate 50→20 cap)
    │       └── HF free tier 도달 시 graceful degrade (skip, dense+sparse RRF only)
    ↓
[Step 9] MMR/diversity 중복 제거
    ↓
[Step 10] top-K 결과 표시
    │       - chunk + page/section + 표 caption + 그림 caption
    │       - 출처 배지 + 매칭 강도 (개발 모드만)
    │       - extractive summary 자동 (BGE-M3 cosine 매칭 강조, 무료)
    │
    └── (사용자 "답변 생성" 클릭 시) → Gemini Flash paid 답변
    ↓
[Step 11] (cross-doc 또는 confidence 낮을 때) "더 자세한 답변 생성" CTA 강조
```

**default on**: Step 4 (query expansion), Step 6 (doc-RRF), Step 8 (reranker), Step 9 (MMR), Step 10 의 extractive summary
**gated**: Step 3a (paid decomposition — complex intent 또는 confidence 낮을 때만)
**조건부**: Step 11 의 paid 답변 (사용자 클릭)
**보류**: HyDE (off 유지)

## 8. 골든셋 draft 샘플 (사용자 작업 가이드 + 12 샘플)

### 8.1 사용자에게 — 어떻게 작성할지

> **솔직하게 적어**. "그때 본 자료에서 X 가 뭐였더라" 같은 톤이 정확. LLM 이 만든 "시험 문제" 톤보다 페르소나 A 의 진짜 query 와 가까움.
>
> 다음 4가지 정도만 신경 써:
> 1. **사람이 자연스럽게 던지는 톤** — "어디 있더라", "그때 봤던", "왜 그런 거지", "이 자료에 있나" 같은 추상 + 구체 조합
> 2. **추상도 다양하게** — 어떤 건 "그때 그 표 뭐였지" 같은 흐릿한 기억, 어떤 건 "쏘나타 디 엣지 인스퍼레이션 가격" 같은 명확한 팩트
> 3. **답이 없을 query 1~2개 포함** — "이 자료에 환경 인증 절차 있어?" 같이 실제 자료에 없는 걸 묻는 케이스 (negative, 환각 검증용)
> 4. **여러 자료를 엮는 query 1~2개** — "운영내규랑 직제규정에서 위원회 역할 어떻게 달라?" 같이 두 doc 비교 (cross_doc)
>
> chunk 번호·정답 페이지는 **몰라도 OK**. `must_include` 컬럼에 "이 단어들이 답에 들어가야 함" 정도만 써. 시스템이 검색 돌려보고 `relevant_chunks` 자동 보강.

## 11. 한 문장 요약

> Claude·Codex 5 라운드 끝에 합의 — **유료 LLM 을 검색 전체에 깔지 않고 cheap signal 로 필요성 판정 후 쓰는 계단형 파이프라인** (cheap router → gated decomposition / vision_need_score → page cap + budget). **S0 비용 계측·캐시·cap 우선** → S1 골든셋 + 실 query 로그 → S2 vision 선별 → S3 검색 path → S4 인제스트 보조 인덱스 → S5 답변 UX. 사용자 결정 (B 기억 보조 + 정확도 80% + Q8 답변 사용자 액션) 와 일관, 페르소나 A 의 빠른 기억 보조 + 비용 예측 가능성 + 복잡 질문 대응 + 출처 신뢰성 4가지 가치 동시 달성.
