## 1. 문제 정의

### 1.1 타겟 문제
한국 직장인(특히 공공기관·대기업 비-IT 실무자)은 하루에 다양한 포맷(HWP, PDF, PPT, 이미지)의 자료를 메신저·이메일로 20건 이상 받지만, **일주일 뒤에는 무엇을 받았는지도, 어디에 있는지도 기억하지 못한다.**

### 1.2 기존 도구가 못 푸는 이유
| 고통 | 기존 도구의 한계 |
|---|---|
| 자료가 여러 메신저/포맷으로 흩뿌려짐 | 노션·에버노트는 수동 정리 부담이 큼 |
| 지나면 내용이 기억 안 남 | 키워드 매칭만 가능, 내 말로 질문 불가 |
| 예전 자료 위치를 모름 | 태그를 직접 달아야 함 |
| HWP/이미지 속 내용은 검색 불가 | 대부분 도구가 한국어 HWP·이미지 OCR에 약함 |
| 외부 클라우드 툴 자체가 차단됨 | 공공·대기업 보안 정책 |

### 1.3 한 줄 가치 제안
> **"정리하지 않아도, 기억의 단편으로 꺼내 쓰는 개인 지식 에이전트."**

## 2. 페르소나 A

### 2.1 프로필
- **A대리**, 33세
- 공공기관 또는 대기업 기획팀, 비-IT 업무
- 주 문서 포맷: **HWP/HWPX** (필수), PPT, PDF, 회의 사진
- 하루 수신량: 메신저·이메일로 20건 이상

### 2.2 사용 환경
| 구분 | 기기 | 비중 |
|---|---|---|
| 업무 PC | Windows | 공공 95%+, 대기업 80%+ |
| 개인 PC | Windows 60~70% / macOS 20~30% | 둘 다 지원 |
| 개인 휴대폰 | Android 약 70% / iOS 약 30% | 둘 다 지원 |

### 2.3 제약 사항
- 업무 PC: 외부 SW 설치 불가, 외부 클라우드 차단
- 업무 자료 외부 반출: 회사 규정상 제한
- 그룹웨어 내장 검색: 품질 열악 (파일명 매칭 위주)

### 2.4 제품 포지셔닝 주의선
- **"업무 DB 대체"가 아닌 "개인 기억 보조"** — 법무/보안 리스크 회피를 위한 필수 선
- 업무 기밀·영업비밀이 섞인 자료는 명시적으로 경고하거나 거부 (§11.3 참조)

## 6. Golden Path 3종

페르소나 A의 실제 환경이 여러 플랫폼이므로 시나리오도 3개.

### 6.1 Primary — "월요일 아침 10분, Windows 업무 PC"

**상황**: 월요일 09:00 팀 회의 5분 전. 임원이 "지난달 기재부 가이드라인 변경점 3분 브리핑" 지시. 08:50.

| 시각 | 행동 | 작동 컴포넌트 |
|---|---|---|
| 08:50 | 브라우저에서 자연어 검색 "지난달 기재부 가이드라인" | T4 쿼리 파싱 → T3 하이브리드 + 메타 |
| 08:51 | `#예산편성` 태그로 결과 좁히기 | T3 메타 필터 재정렬 |
| 08:53 | 45p HWP 탭 → 사전 계산된 요약·변경점 확인 | T1 인제스트 시점 사전 요약 |
| 08:55 | "작년 대비 바뀐 부분만" 단일 문서 QA | T4 스코프=doc_id, 인용 포함 생성 |
| 08:58 | 출처 배지 `[p.12]` 클릭 → 원문 하이라이트 | chunk↔page 매핑 |
| 09:00 | "브리핑 노트 내보내기" → 회의 입장 | — |

**입력 경로**: 데스크톱 드래그앤드롭, 키보드 단축키 기반.

### 6.2 Secondary — "퇴근길, Android 지하철"
카톡으로 받은 HWPX → **Android 공유시트로 앱 전송** → S6 진행 → 집 도착 전 3줄 요약 훑기 → 내일 회의 태그 붙이기.

### 6.3 Tertiary — "아이폰, 카페에서"
팀 슬랙의 외부 레퍼런스 PDF → **iOS Shortcuts "내 기억 저장" 단축어**로 전송 (iOS Safari PWA의 공유시트 제약 우회) → S6 진행 → 카페에서 요약 읽기.

### 8.3 차별점 3개

1. **"한국 직장인의 문서 현실"에 맞춘 유일한 도구** — HWP/HWPX + 공공·대기업 보안 맥락 + 한국어 RAG 셋다 되는 제품 없음
2. **"출처가 증명되는 답변"** — chunk→page→문단 매핑을 처음부터 설계에 박음
3. **"업로드 시 일해두는 제품"** — 검색 시점이 아니라 인제스트 시점에 요약·태그·변경점 생성

### 8.4 포지셔닝 문장
- 실무 톤: "HWP까지 읽는 한국어 RAG 기반 개인 지식 에이전트"
- 감성 톤: "정리하지 않아도, 기억의 단편으로 꺼내 쓰는 앱"
- 기술 톤: "하이브리드 검색 + 출처 추적 + 로컬 교체 가능한 어댑터 설계"

### 8.5 발표 내러티브 3막
- **1막 문제**: 한국 직장인의 하루 — HWP 20개, 카톡 스샷 30장, 일주일 뒤 망각. 기존 도구는 차단·한국어 약함·HWP 미지원
- **2막 선택**: 3가지 결정 — HWP 1등 시민, 하이브리드 검색, Cloud→Local 어댑터
- **3막 증명**: Ragas Faithfulness N / HWP 인제스트 N% / 출처 일치율 N% / P95 응답 Ns. Golden Path 1분 데모

### 8.6 예상 반박 대응
| 공격 | 대응 |
|---|---|
| Notion AI가 HWP 지원하면? | 경쟁자는 글로벌 타깃, HWP는 한국 전용 → **타이밍의 해자** |
| 결국 LLM 래퍼? | T1~T5는 LLM 밖 엔지니어링 — 어댑터 구조 + 평가 루프 시연 |
| 로컬도 안 되면서 보안? | 어댑터 레이어로 교체 가능 (§9.4). MVP는 Cloud, v2 Local 로드맵 |
| 평가 과적합? | 평가셋 공개, US-01~09 기반 시나리오 벤치 |
| 스샷 OCR 오독으로 거짓 답변? | UI 신뢰도 등급 + 숫자 답변에 원문 확인 배지 (§12.3) |

1. MVP는 전부 **무료 티어**로 돌아가야 한다

### 9.3 핵심 선택 근거

**BGE-M3 임베딩**: 한 모델에서 dense+sparse 동시 출력 → 별도 BM25 엔진 불필요. 한국어 포함 다국어 학습, Apache 2.0.

**BGE-M3 로컬 실행 대신 HF Inference API**: Railway 무료 512MB RAM 제약 회피. 어댑터 구조는 동일.

**Gemini 2.0 Flash**: 무료 티어(일 1,500회), 한국어 품질 충분, Vision 내장 → T2 1-pass 처리. MVP 비용 0원 유지.

**HWPX 크리티컬 패스 고정**: 2010+ 포맷이 80%+. XML 파싱 난이도 낮음. HWP 구포맷 실패는 graceful degradation.

## 10. 인제스트 파이프라인 (D)

### 10.1 설계 원칙
1. 업로드 시점 투자 = 검색 시점 절약
2. 부분 성공 허용
3. 모든 외부 API retry/backoff (최대 3회, 지수 대기)
4. Idempotent (SHA-256 + 임베딩 유사도)
5. UI에 모든 단계 투명 표시

### 10.2 전체 플로우

```
수신 경로 5종
      ▼
[1] 입력 게이트 (파일 수준) — 거부/통과
      ▼
[2] 원본 저장 (Supabase Storage, SHA-256 dedup)
      ▼
[3] 작업 큐 등록 — 동기 응답 "접수됨" < 2초
  ═══════════ 비동기 워커 ═══════════
      ▼
[4] 포맷별 추출 (PDF/HWPX/HWP/DOCX/IMG/URL)
      ▼
[5] Vision 캡셔닝 (이미지·표·도표)
      ▼
[6] 입력 게이트 (콘텐츠 수준) — 플래그 부여
      ▼
[7] 청킹 (섹션 기준 → semantic → 병합)
      ▼
[8] 자동 태그 · 사전 요약 · 변경점 diff (Gemini Flash 1~3회)
      ▼
[9] 임베딩 (BGE-M3 dense + sparse)
      ▼
[10] 적재 & 인덱싱 (pgvector + JSONB)
      ▼
[11] 중복/유사 감지 (Tier 1~3)
      ▼
[12] 완료 알림 (WebSocket / PWA 푸시)
```

### 10.3 포맷별 추출 세부

| 포맷 | 도구 | 실패 대응 |
|---|---|---|
| PDF | PyMuPDF + Camelot | 스캔본 감지 → 이미지 경로로 재라우팅 |
| HWPX | python-hwpx | — |
| HWP 구포맷 | hwp5html CLI | 실패 시 "지원되지 않는 버전" 메시지, graceful 종료 |
| DOCX | python-docx | — |
| 이미지 | Gemini Vision 단독 | HEIC → pillow-heif 변환 선행 |
| URL | trafilatura | — |

### 10.4 Vision 캡셔닝 프롬프트 (단일 호출, 4가지 동시 요청)

```
이 이미지를 분석하세요. JSON으로 응답:
{
  "type": "문서|스크린샷|메신저대화|화이트보드|명함|차트|표|기타",
  "ocr_text": "읽을 수 있는 모든 텍스트, 원문 그대로",
  "caption": "이미지 내용을 1~2문장",
  "structured": { /* 표→markdown, 차트→축/값 */ }
}
```

`type == "메신저대화"` → 입력 게이트에서 `flag_third_party` 설정.

### 10.5 청킹 전략
- **1차**: 섹션/헤딩 기준 분할
- **2차**: 800자 초과 섹션은 LlamaIndex `SemanticSplitterNodeParser`
- **3차**: 200자 미만은 인접 섹션과 병합
- 최대 1,000 토큰, 섹션 경계 넘는 경우만 100자 오버랩
- 메타 보존: `chunk_idx`, `doc_id`, `page`, `bbox`, `section_title`, `char_range`

### 10.6 자동 태그·요약·diff

**태그 (호출 1)**: 문서 앞 3,000자 → JSON (topic_tags, entity_tags, document_type, time_reference)

**요약 (호출 2)**: map-reduce → `summary_3line` + `implications` (페르소나 프로파일 주입)

**변경점 diff (호출 3, 조건부)**: 파일명 유사도 0.6+ AND 문서 임베딩 유사도 0.85+ 시 트리거

### 10.7 저장 스키마

- `documents`: id, user_id, title, doc_type, tags[], summary, implications, doc_embedding, flags, created_at, source_channel
- `chunks`: id, doc_id, chunk_idx, text, page, bbox, section, dense_vec, sparse_json
- `ingest_jobs`: id, status, current_stage, started_at, error_msg
- `ingest_logs`: job_id, stage, started_at, finished_at, error_msg
- 인덱스: `ivfflat` on dense_vec, GIN on tags[], GIN on sparse_json

### 10.8 중복 감지 3-Tier

| Tier | 방법 | 결과 |
|---|---|---|
| 1 | SHA-256 일치 | 즉시 "이미 저장됨", 업로드 취소 |
| 2 | 문서 임베딩 유사도 ≥ 0.95 | "거의 동일한 자료" 알림, 머지 제안 |
| 3 | 파일명 유사 + 임베딩 ≥ 0.85 | "이전 버전 관계 추정", 변경점 diff 트리거 |

### 10.9 비동기 아키텍처

- MVP: FastAPI `BackgroundTasks` (단일 프로세스)
- 동시 처리 한도: 3건
- 진행 상태: `ingest_jobs` 테이블, WebSocket 또는 폴링
- v2: `rq` + Redis 분리

### 10.10 실패·재시도

| 단계 | 실패 시 |
|---|---|
| 게이트 | 즉시 사용자 알림, 저장 X |
| 원본 저장 | 3회 retry → 실패 알림 |
| 포맷 파싱 | 부분 실패 허용 (페이지 스킵) |
| Vision | 이미지별 3회 retry → "미처리" 플래그, 다른 단계 진행 |
| 태그·요약 | 3회 retry → 저장하되 필드 null |
| 임베딩 | 3회 retry → 실패 시 전체 롤백 |
| 적재 | 3회 retry → 큐 재시도 |

실패한 job은 S6에서 "다시 시도" 버튼 노출.

## 12. 스샷 처리 정책 (H)

### 12.1 스샷 유형별 품질 현실

| 유형 | 품질 | 주요 리스크 |
|---|---|---|
| 카톡/메신저 스샷 | 높음 | 개인정보 (제3자 대화) |
| PDF 문서 부분 스샷 | 중상 | 원본 중복 |
| 웹페이지 스샷 | 중상 | URL 수집이 더 우수 |
| 엑셀/표 스샷 | 중하 | 숫자 오독 |
| PPT 슬라이드 사진 (회의실) | 하 | 각도·반사·조명 |
| 손글씨 / 화이트보드 | 매우 하 | Vision 환각 |
| 작은 글씨 스샷 | 하 | 해상도 손실 |

### 12.2 기획상 리스크 5가지
1. 메타데이터 부재 → 쿼리 파싱 기반 약화
2. 파편화 — 연속 스샷 클러스터링 필요
3. 원본과 스샷 공존 → 중복 감지 복잡
4. 환각 위험 — US-09와 정면 충돌 (12% → 17% 오독)
5. 제3자 정보 (카톡 타인 대화) → 개인정보보호법 이슈

### 12.3 "2등 시민" 포지셔닝 — 핵심 결정

- 원본 파일이 있으면 원본 우선
- 스샷은 보조 자료
- UI에 신뢰도 등급 명시:
  - `✅ 원본 문서`
  - `⚠️ 스샷 기반 (숫자·표는 부정확할 수 있음)`
- 답변에 숫자가 포함되고 출처가 스샷이면 `🔍 원문 확인 권장` 배지

### 12.4 "원본으로 교체" 기능 (v1.5)
스샷 업로드 후 원본을 찾아 올리면 내용 유사도로 자동 매칭 → 사용자 확인 → 스샷은 "임시 기억"으로 강등.

### 12.5 MVP 범위
- **포함**: 메신저·문서 스샷 (품질 상·중상)
- **v2 이관**: 화이트보드·PPT 사진·손글씨 (품질 하)
- **범위 외 선언**: 카톡 단톡방 스샷은 §9.7 제3자 카테고리로 매번 경고
