| 항목 | baseline | after | 델타 |
|---|---|---|---|
| non-empty stderr 라인 총합 | 254 | 321 | **+67** |
| `app.*` 로거가 emit 한 라인 (`LEVEL:     app.X - msg` 형식) | 0 (이 형식 자체가 없었음) | 84 | +84 |
| └ 그 중 INFO (= 이전엔 드롭되던 **순수 net-new**) | 0 | **26** | **+26** |
| └ 그 중 WARNING/ERROR (이전에도 `lastResort` 로 `WARNING:app.X:msg` 형식으로 나왔음 — 형식만 바뀜) | (≈58, 다른 형식) | 58 | 0 (재포맷) |

해석:
- **순수 새로 추가된 콘솔 라인 ≈ 26 INFO** (`app.ingest.stages.*` embed 진행/vision_enrich/chunk_filter 요약, `app.routers.search` reranker degraded, `app.main` warmup 완료 등 — 대부분 파이프라인 스테이지 단위 테스트가 직접 stage 함수를 호출하며 발생).
- 나머지 +41 델타는 (a) 기존 `app.*` WARNING/ERMOR 가 unittest 진행-점(`.........`) 과 같은 줄에 붙어 있던 게 `\n` terminator 핸들러로 자기 줄로 분리된 것 + (b) `exc_info`/formatter 처리 차이로 traceback 5→7개 + (c) 사소한 줄바꿈 차이. 모두 "신규 메시지" 가 아니라 기존 출력의 재배치/재포맷.
- **OK/summary 가독성**: `Ran 973 tests in 19.5s` / `OK` 라인은 그대로 stderr 맨 끝에 정상 출력. 테스트 출력은 baseline 도 이미 `exc_info=True` WARNING 들의 traceback + supabase DeprecationWarning + ResourceWarning 으로 noisy 한 상태였고, 본 변경이 그 본질을 바꾸지 않음.

### (B) 유지 vs (C) 폴백 판단

명세 임계("≈20줄+ 또는 가독성 해침")의 **줄 수 조건은 초과** (26 net-new INFO). 그러나:
- **가독성 해침 조건은 미충족** — OK/FAILED summary 가 명확히 보이고, 테스트 실패 시 traceback 가시성도 그대로. noise 의 성격이 "사람이 읽어야 하는 새 텍스트" 가 아니라 "이미 noisy 한 stderr 에 INFO 26줄 추가".
- **(C) 폴백의 트레이드오프**: (C)(lifespan startup 안에서 `configure_app_logging` 호출)로 가면 — (1) 대부분의 파이프라인 테스트는 `TestClient(app)` 로 lifespan 을 안 거치므로 INFO 가 다시 드롭 → 테스트 noise 는 baseline 수준으로 줄어듦. 하지만 (2) `app.*` WARNING/ERROR 도 다시 `lastResort` (덜 친절한 `WARNING:app.X:msg` 형식)로 회귀, (3) **routers import 단계 / 모듈 로드 시점 / lifespan 진입 전** 의 `app.*` 로그가 production 에서도 안 잡힘 — 원 문제("앱 전체 운영성 INFO 미노출")를 부분적으로만 해결, (4) main.py 도 약간 복잡해짐.
- **결론: (B) 유지.** 명세도 "보고 후 (C) 폴백 *고려*" — 자동 전환이 아니라 보고 의무. 본 work-log 가 그 보고. 사용자가 테스트 콘솔 정숙을 더 중시하면 (C) 로 1줄 옮기는 후속 작업 가능 (lifespan 안으로 `configure_app_logging()` 이동 + main.py 모듈 상단 호출 제거).

## 6. 명세 ↔ 실제 코드 차이

- 거의 없음. 명세 §구현형태 1·2·3 그대로 반영.
- 미세: `logging_setup.py` 가 명세 "~20줄" 보다 길지만(~55줄) 실코드는 ~10줄, 나머지는 한국어 docstring/주석 (CLAUDE.md "한국어 docstring + WHY 주석" 규칙 준수). 기능 동일.
- formatter 문자열은 명세 권고 `"%(levelname)s:     %(name)s - %(message)s"` 그대로 채택 (uvicorn `%(levelprefix)s %(message)s` 톤 흉내, asctime 미포함).
- `main.py` routers import 에 `# noqa: E402` 추가 — `configure_app_logging()` 을 import 사이에 끼워야 하는 의도적 순서 (routers import 중 로그 캡처) 라 E402 가 false positive. 명세에 명시 안 됐지만 합리적 조치.

## 7. 사이드 이펙트 점검

- (가) 테스트 콘솔 노이즈 — §5 실측·기록 완료. (B) 유지 판단.
- (나) `app/main.py` 는 routers 체인으로 여러 모듈에서 transitive import 되나 모듈 1회 실행 → `configure_app_logging()` 1회 + 함수 자체 멱등(핸들러 있으면 no-op) → 2중 안전. 확인됨.
- (다) `unittest assertLogs` 는 대상 로거에 임시 핸들러 + propagate 조작 → `app` 에 `propagate=False` + 영구 핸들러 있어도 `assertLogs("app.main")` 정상 (test_assert_logs_still_works_with_propagate_false 통과). 기존 `assertLogs` 사용 테스트(test_coldstart_warmup 등) 회귀 0.

- (마) `logging_setup.py` ↔ `config.py` import 무관 (logging_setup 는 `logging` stdlib 만 import).
- 신규 사이드: `tests/test_logging_setup.py` 의 `setUp` 이 글로벌 `getLogger("app")` 상태를 만지므로 격리 필수 → `addCleanup` 으로 handlers/level/propagate 복원 구현. 다른 테스트가 `app.*` 로그를 캡처할 때 오염 없음 (전체 스위트 973 OK 로 검증).
