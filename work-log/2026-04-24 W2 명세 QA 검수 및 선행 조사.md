### 1.2 DE-20 Railway `hwp5html` 설치 가능성 — **PARTIAL**

**조사 방법**: Railway 공식 문서 + Nixpacks GitHub + PyPI pyhwp 페이지

**결과**:
- **Nixpacks** (W1 기준 기본 빌더) 는 신규 서비스에서 deprecated, Railway 는 **Railpack** 으로 전환 중. 기존 서비스는 계속 Nixpacks 사용 가능
- 시스템 패키지 추가는 `nixpacks.toml` 의 `[phases.setup]` 에 Nix pkg 이름 추가로 가능 (Puppeteer 등 성공 사례)
- **하지만 `hwp5html` (Python pyhwp 패키지) 은 nixpkgs 레지스트리에 없음**. `pyhwp` 는 PyPI 패키지이므로 `pip install pyhwp` 로 설치되고, `hwp5html` CLI 는 패키지가 제공하는 Python entry point
- 즉 **Nix 시스템 패키지가 아니라 Python 의존성으로 처리해야 함** → Railway Nixpacks Python phase 에서 자동 감지 가능 (`pyproject.toml` / `requirements.txt` 에 `pyhwp` 명시 시)

**채택 결정**:
- DE-20 을 **"가능, 단 Python 의존성으로 처리"** 로 확정
- 단 pyhwp 의 Python 3.12 호환성이 확실치 않음 (마지막 릴리스 2020) → Day 1 에 로컬 `uv add pyhwp` 로 설치 시도 → 실패 시 **HWP 구포맷 전면 제외 (DE-13 100%)**
- 명세 §3.F "Railway 설치 가능 여부 Day 1 조사" 는 **로컬 설치 시도로 치환**

### 2.1 CRITICAL (Day 1 전 반드시 해결)

#### A-1. 파이프라인 스테이지 모순

**문제**:
- 기획서 §10.2 플로우 (12 steps, 비동기 워커 [4]~[12]) — `[4]extract → [5]vision → [6]게이트B → [7]청킹 → [8]태그요약 → [9]embed → [10]load → [11]dedup → [12]알림`
- W1 실구현 (7-stage) — `extract → chunk → tag_summarize → load → embed → doc_embed → dedup`
- W2 명세 §3.G — "단계 6 content_gate 삽입 (7→8-stage), 위치: `extract → chunk → content_gate → tag_summarize → ...`"
- **세 문서가 각각 다른 순서**. 기획서는 "게이트 B → 청킹", 명세는 "청킹 → 게이트 B" — 정반대
- Vision 을 별도 스테이지로 둘지 (§10.2 [5]), `ImageParser` 내부로 흡수할지 (명세 §3.D) 도 미결정

**영향**: 파이프라인 stage 상수 · `ingest_jobs.current_stage` · 프론트 `STAGE_ORDER[]` · `STAGE_LABELS{}` · `StageProgress` 컴포넌트 모두 영향. 7→8 또는 7→9 stage 확장은 W1 타입/DB breaking change

**해결안**:
Day 1 전 senior-planner 에 3가지 결정 요청:
1. content_gate 위치 — "chunk 이후" (명세 현행) vs "chunk 이전" (기획서 §10.2)
2. Vision 분리 — 별도 stage vs `ImageParser` 내부 composition
3. W2 최종 stage 리스트 (7 → 8? 9?)
- 확정 후 명세 §8 갱신 요청에 **전체 스테이지 리스트 박아넣기** 필요
