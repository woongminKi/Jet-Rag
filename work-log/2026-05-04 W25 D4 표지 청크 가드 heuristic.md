## 배경

W25 D3 마감 후 사용자 직접 발견한 검색 품질 결함:

- **검색어**: `소나타에서 제공하는 시트 종류 뭐가 있어?`
- **현상**: top-1 doc=sonata 카탈로그, top-1 chunk=p.1 표지 (`chunk_idx=0`, text="SONATA" 6자)
- **원인 (이전 senior-qa SQL 검증)**:
  - p.1 chunk_idx=0 의 text="SONATA" (6자), flags={}
  - PGroonga sparse 매칭 0건 (한국어 토크나이저 실패) → dense (BGE-M3) 단독 ranking
  - dense cos sim 이 짧은 표지 청크와 한국어 자연어 사이 비정상 우세
- **사용자 판단**: "이런 지점은 걸러야 할 것 같아"

## 결정 — 단순 후처리 패널티 (안 a)

**위치**: `app/routers/search.py` — RPC 결과 직후, `doc_chunk_scores` 누적 단계.

**조건 (보수적, 모두 만족)**:
- `text_len <= 30` (극단 짧은 청크)
- AND (`chunk_idx == 0` OR `page == 1`) — 표지 영역 추정

**처리**: `rrf_score *= 0.3` (`_COVER_GUARD_PENALTY`)

**상수**:
- `_COVER_GUARD_TEXT_LEN = 30`
- `_COVER_GUARD_PENALTY = 0.3`

**비용**: chunks IN-list 쿼리 1회 추가 (max 200 chunk_id) — latency 영향 미미 (warm 200ms 내외).

## 대안 안 trade-off

| 안 | 위치 | 비용 | 효과 | 회귀 risk |
|---|---|---|---:|---|
| **(a) search.py 후처리 패널티 (채택)** | RPC 결과 후 score 곱셈 | chunks fetch 1회 | 즉시 | 짧은 헤딩 false positive — chunk_idx>0 AND page>1 조건으로 회피 |
| (b) chunk_filter 의 is_cover 마킹 + RPC 가드 | chunk_filter.py + 새 SQL 마이그레이션 | 중간 | 영구적 | chunks.flags 컬럼 backfill 필요 |
| (c) dense_rank==1 + text_len<30 동시 패널티 | RPC schema 변경 | 중간 | 가장 정확 | RPC SQL 변경 — 008 fused CTE 가 dense_rank 노출 여부 확인 |

(a) 채택 근거: 빠른 ship + 효과 측정 → (b)/(c) 결정 데이터 확보.
