## 1. 오늘 달성한 것

- `google-genai>=1.0` (해결 1.73.1) — Gemini 호출

### 1.2 로그 인터페이스 리팩토링 (커밋 2)
Day 3 E2E 에서 발견된 `started_at > finished_at` 역전 해소.

| Before | After |
|---|---|
| 한 스테이지에 `log_stage(started)` + `log_stage(succeeded)` 2행 | `begin_stage()` INSERT 1행 → `end_stage()` UPDATE 로 마감. `stage()` context manager · `skip_stage()` 헬퍼 추가 |

스키마 변경 없음. Python 인터페이스만 재구성.

### 1.3 PyMuPDF 어댑터 (커밋 3)
- `PyMuPDFParser` — `DocumentParser` Protocol 구현. `source_type='pdf'`
- Protocol 시그니처 변경: `parse(file_path)` → `parse(data: bytes, *, file_name)`
  - 이유: 업로드 bytes → Storage → 재다운로드 → parse 흐름에서 임시 파일 I/O 불필요
- `fitz.open(stream=data, filetype='pdf')` 로 메모리 직접 파싱
- `page.get_text("blocks")` 로 블록 단위 추출 (`(x0,y0,x1,y1,text,block_no,block_type)`)
- 페이지 단위 부분 실패 허용 (`warnings` 누적), 전체 fail 아님
- 섹션 제목 감지·스캔본 감지는 W2 이후

### 1.4 extract 스테이지 (커밋 4)
`ingest/stages/extract.py` — `run_extract_stage(job_id, doc_id) -> ExtractionResult | None`

- PDF: Storage 다운로드 → PyMuPDFParser → `ExtractionResult` 반환
- 비 PDF (`.md`, `.docx`, …): `documents.flags.extract_skipped=true` + `ingest_logs.status=skipped` + `None` 반환 → pipeline 이 후속 스테이지 건너뛰고 `finish_job`
- 분기 로직을 스테이지 내부에 흡수해 pipeline 은 반환값만 본다

### 1.5 chunk 스테이지 (커밋 5)
`ingest/stages/chunk.py` — 기획서 §10.5 3단계 청킹

1. **1차**: PyMuPDF 가 블록 단위로 이미 분할해 준 섹션 그대로
2. **2차**: 1,000자 초과 섹션을 문장 경계로 800자 근방 분할
   - 정규식: `r"(?<=[.!?])\s+ | (?<=[다요까죠습니다])[.!?]?\s+ | \n\s*\n"` — 한국어 종결어미 + 영문 + 문단 break
   - 문장 경계 못 찾으면 강제 800자 분할
3. **3차**: 200자 미만 섹션을 같은 페이지 인접과 병합 (최대 1,000자까지)

메타 보존: `chunk_idx` · `doc_id` · `page` · `section_title` · `bbox` · `char_range`

**ChunkRecord 조정** (커밋 5 포함)
- `dense_vec: list[float] → list[float] | None` — Day 4 엔 None, Day 5 에 UPSERT 로 채움
- `sparse_json` 기본 빈 dict
- `bbox` 필드 Day 3 에 누락됐던 것 추가
- `SupabasePgVectorStore._serialize_chunk` 를 None 필드 생략 모드로 조정

### 1.6 load 스테이지 + pipeline 전반부 결합 (커밋 6)
`ingest/stages/load.py` — `SupabasePgVectorStore.upsert_chunks` 호출

Pipeline: `extract → chunk → load` 순차. `extract` 가 `None` 이면 조기 `finish_job`. Router 는 `add_task(run_pipeline, job.id, doc_id)` 로 doc_id 도 전달.

Day 4 전반부(커밋 1~6) 시점 E2E:
- 3p fitz-생성 PDF (한글 글리프 깨진 테스트용) → 1.3초 수신 · chunks 3행 · 태그·요약은 아직 없음

### 1.7 Gemini 2.0 Flash 어댑터 (커밋 7)
`GeminiLLMProvider` — `LLMProvider` Protocol 구현
- ChatMessage(system/user/assistant) → Gemini Content(system_instruction + parts)
- JSON 모드, 이미지 첨부(W2 Vision 용)
- 3회 retry + 지수 백오프 + jitter (§10.10)

### 1.8 tag_summarize 스테이지 (커밋 8)
`ingest/stages/tag_summarize.py` — 기획서 §10.6 중 태그·요약 (diff 는 Day 5)

- **태그 호출** (앞 3,000자): JSON `{topic_tags, entity_tags, document_type, time_reference}`
- **요약 호출** (앞 12,000자): JSON `{summary_3line, implications}`
- 실패 정책(§10.10): 내부 retry 후 최종 실패 → 예외 swallow, `documents` 필드 NULL 유지, 파이프라인 계속
- 부분 성공 허용 (태그 성공 / 요약 실패)
- 응답 JSON 이 코드블록으로 감싸져도 방어 파싱

### 1.9 pipeline 최종 결합 (커밋 9)
`extract → chunk → tag_summarize → load`. tag_summarize 실패는 파이프라인 중단 X.

### 1.10 Gemini 모델 교정 (커밋 10·11)

## 2. 실 한국어 PDF E2E 결과

테스트 대상: `assets/sample-report.pdf` — 한국은행 경제전망 리포트 8.6MB, 93 페이지 (사용자 준비).

### 2.1 소요 시간

| 단계 | 시간 |
|---|---|
| 수신 응답 (HTTP 202) | 3.70초 |
| extract | 2,699 ms |
| chunk | 1 ms |
| tag_summarize | ~27,000 ms (Gemini 503 1회 retry 포함) |
| load | 320 ms |
| **총 파이프라인** | **~33초** |

**SLO 비교** (기획서 §10.11)
- 수신 <2초: **살짝 초과** (8.6MB · Storage 동기 업로드 때문). 소형 파일(<1MB)에서는 이미 Day 3 에서 <2초 확인. W2 에 Storage 업로드를 BackgroundTask 로 떼어 분리 고려
- PDF 50p <60초: ✓ 93p 에서도 33초

### 2.2 파이프라인 산출물

```
documents
├── tags (17) : [경제전망, 경제성장, 물가, 고용, 세계경제, 반도체 경기, 리스크 평가,
│                Indigo Book, 미국, 유로지역, 중국, 일본, 아시아 신흥국, AI, 브렌트유,
│                금융통화위원회실, OBBBA]
├── flags.document_type : 보고서
├── flags.time_reference: 2026-02
├── summary  : 2026년 한국 경제는 반도체 호조와 세계 경제 흐름에 힘입어 2.0% 성장 전망.
│              물가는 2.2% 상승, 경상수지 흑자 확대 예상되나, 美관세와 건설 부진이 제약 요인.
│              AI 투자와 정책 완화가 성장을 견인하지만, 통상 불확실성과 비IT 부문 온도차는 리스크.
└── implications: AI 및 반도체 중심의 수출 호조가 경제 성장을 견인하지만, 미국 관세와
                  비IT 부문 부진 등 대내외 리스크가 상존함을 보여준다. (이하 생략)

chunks: 375 rows, pages 1~93 전체 커버
```

**품질 평가 (정성)**
- 태그·문서유형·시점 모두 정확. OBBBA 같은 약어까지 entity_tags 로 보존
- 3줄 요약은 KDI 보고서 핵심을 정확히 반영: 반도체 호조·미국 관세·AI 투자
- implications 는 "개인 지식 관점" 맥락으로 재해석 — 프롬프트 의도 반영 성공

### 2.3 Gemini 호출 현황

- 호출 3회 (smoke 1 + 태그 1 + 요약 1) → RPD 3/20 소비
- 503 UNAVAILABLE 1회 → 자동 retry 로 성공 (§10.10 정책 실전 검증)

```
│   ├── vectorstore.py              ← ChunkRecord 에 bbox 추가, dense_vec Optional

│       ├── gemini_llm.py           ← NEW — Gemini 2.5 Flash LLMProvider

│       └── supabase_vectorstore.py ← None 필드 생략 serialize

    ├── jobs.py                     ← begin_stage / end_stage / stage() / skip_stage
    ├── pipeline.py                 ← extract → chunk → tag_summarize → load
    └── stages/                     ← NEW 패키지
        ├── __init__.py
        ├── extract.py
        ├── chunk.py
        ├── tag_summarize.py
        └── load.py
```

파이프라인 실행 경로:
```
POST /documents
  ↓ (입력 게이트 A + SHA-256 dedup + Storage 업로드 + documents insert)
BackgroundTasks.add_task(run_pipeline, job_id, doc_id)
  ↓
extract → chunk → tag_summarize → load → finish_job
```

## 5. 주요 의사결정 스냅샷 (오늘 추가분)

| 항목 | 값 | 근거 |
|---|---|---|
| Parser Protocol | `parse(bytes, *, file_name)` | 임시 파일 I/O 우회, Storage bytes 직접 소비 |
| 비 PDF 처리 | graceful skip + flags 마킹 | W2 멀티포맷 어댑터 붙이면 reingest 로 자동 처리 |
| 청킹 전략 | 1000자 상한 + 한국어 종결어미 휴리스틱 | LlamaIndex 미도입 — 의존성 최소화, 커스터마이즈 용이 |
| Gemini 모델 | `gemini-2.5-flash` | 2.0-flash 는 2026-04 기준 유료 전용. 무료 등급 = 2.5-flash |
| tag_summarize 실패 정책 | 예외 swallow + 필드 NULL | 기획서 §10.10. 태그·요약 실패가 파이프라인 전체 실패가 되지 않아야 함 |
| 로그 행 모델 | 1행 (begin → end update) | Day 3 `started_at > finished_at` 역전 해소 |
| `dense_vec` 스토리지 | NULL 허용 | Day 5 임베딩 스테이지가 동일 `(doc_id, chunk_idx)` 키로 UPSERT 하여 채움 |
