## 2. 비판적 재검토 (사용자 §1)

### 2.1 옵션 비교

| 옵션 | 설계 | 장점 | 단점 |
|---|---|---|---|
| **A** | in-memory module-level counter + threading.Lock | search_metrics 패턴 재사용, stdlib only, 의존성 0 | 프로세스 재시작 시 휘발 |
| B | DB 영속화 (`vision_usage_logs` 테이블) | 정확성↑, 영구 추적 | 마이그레이션 + 성능 영향 |
| C | 외부 메트릭 (Prometheus) | 운영 표준 | 의존성↑, 인프라 복잡도↑ |

**결정 — A 채택**. MVP 단일 사용자 단일 worker 환경에서 in-memory 충분. search_metrics 와 동일 휘발성 trade-off 일치.

- **PPTX rerouting** (W8 Day 2 신규)

`GeminiVisionCaptioner.caption` 직접 호출 위치는 ImageParser 한 곳뿐 → 단일 record 지점으로 충분.

## 3. 구현

### 3.1 `app/services/vision_metrics.py` 신규

```python
def record_call(*, success: bool) -> None: ...
def get_usage() -> dict:  # {total_calls, success_calls, error_calls, last_called_at}
def reset() -> None:  # 테스트용
```

- `threading.Lock` 으로 thread-safe
- `last_called_at` UTC timezone-aware ISO 8601

### 3.2 `app/adapters/impl/image_parser.py` 통합

```python
try:
    caption = self._captioner.caption(normalized_bytes, mime_type=normalized_mime)
except Exception:
    vision_metrics.record_call(success=False)
    raise
vision_metrics.record_call(success=True)
```

raise 도 카운트 → error_calls 분리 추적 (RPD 초과 vs API 일시 오류 구분 가능).

### 3.3 `/stats` 응답 확장
