## 3. 핵심 발견 (다음 세션이 알아야 할 사실)

### 3.1 vision_page_cache 사전 cap check 의 historical SUM 차단 = M2 W-4 회귀 직접 원인 (P1)

- `documents.flags.vision_budget.used_usd` = `vision_usage_log` doc_id SUM (**historical 누적**, 전 reingest 합).
- 재인제스트 시 사전 cap check (extract.py:220 / incremental.py:440) 가 cache lookup 전에 historical SUM 보고 차단 → vision OCR skip → vision_chunks 0.
- inner-loop check 는 cache hit 시 `vision_metrics.record_call` 호출 0 → SUM 증가 0 → 통과 가능 (정상).
- **Fix (P1, `57d87e4`)**: `count_uncached_pages` 헬퍼로 모든 페이지 cache hit 시 사전 check 우회. graceful (count None / fitz 실패 / sha256 None → 보수적 fallback).

### 3.2 KPI #7 단순 R@10 vs qtype-aware top-1 (DECISION-13)

```
overall (단순):
  hybrid  R@10 0.6817 / top-1 0.7797 / P95 174ms
  dense   R@10 0.7137 / top-1 0.7797 / P95 157ms  ← R@10 +3.2pp 우세
  sparse  R@10 0.5268 / top-1 0.5593 / P95 129ms
```

**단순 R@10 +5pp 기준은 ❌**. but **qtype 별**:
- hybrid 압도: table_lookup top-1 **+0.34** / numeric_lookup **+0.43** / cross_doc R@10 **+0.045**
- dense 압도: 일반 nl query 의 broad R@10

→ **DECISION-13 채택**: KPI #7 = qtype-aware top-1 (table+numeric) AND cross_doc R@10. 도메인 가치 큰 query type 의 사용자 체감 정확도 직결. 운영 default = hybrid 유지 (DECISION-1 정합).

### 3.3 S5-A backend↔frontend signal 키 mismatch dead code 회귀

- backend `intent_router.IntentRouterDecision.triggered_signals` = `T1_cross_doc`, `T2_compare`, ..., `T7_multi_target` (7종, **T 접두사**).
- frontend `RouterSignalsBadge` SIGNAL_META = `cross_doc`, `temporal`, `ambiguous` (3종, **접두사 없음**).
- 키 mismatch → graceful filter 가 모든 signal skip → badge **0회 발화** (dead code).
- **Fix (`da80b86`)**: SIGNAL_META 키 backend 정합 (`T1_cross_doc`, `T7_multi_target` 2종만 노출, T2~T6 graceful filter). + `decomposed_subqueries` `/search?q=...` CTA 추가.

### 3.4 acceptable_chunks 추가 → R@10 정직 측정 효과

- Acceptable judge 2차 (`0c5d3e0`): 7건 채움 → golden_v2 acceptable_chunks 확장.
- R@10 0.6817 → 0.6738 (−0.0079) / top-1 0.7797 동일.
- **직관 의외 해석**: graded recall = ∑(회수·score) / ∑(정답·score) → 정답 set 분모 ↑ → R@10 약간 감소. 단순 회귀 아닌 **정직 측정** (DECISION-6 정신 연장, 기획서 §13.3 정합).

### 3.5 G-U-022 outlier — retriever + judge 양쪽 약점

- query: "스튜디오앤드오어 로고 디자인 컨셉이 뭐야" (fuzzy_memory)
- RAGAS: Faithfulness 0/0 (KPI #9 환각률 9.2% 의 주 outlier)
- Acceptable judge 2차: empty (candidate top-30 + threshold 0.6 미달)
- /search 직접 호출: 정답 doc `브랜딩_스튜디오앤드오어` 가 **4위** (1~3위 = 다른 doc), chunk 3 회수됨.
- **원인**: 고유명사 (`스튜디오앤드오어`) 의 dense embedding 약점 + sparse signal 미흡 (cross-doc relevance score 가 다른 doc 가 더 높게 점수).
- **fix 영역**: v1.5 (self-host BGE-M3 + ColBERT / query expansion 강화 / golden_v3 분포 시정).

### 3.6 KPI #11 인제스트 SLO 48.3% 미달 — HF cold-start 직접 원인

- 28/58 = 48.3% (게이트 90% 미달, -41.7pp)

- 원인: HF BGE-M3 free-tier scale-to-zero cold-start (~30~67s, batch 마다 반복) + vision enrich 페이지 latency
- **fix 영역**: v1.5 (HF self-host + vision_page_cache 적극 활용 + 분할 재인제스트)
