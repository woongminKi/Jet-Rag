## 1. 설계 결정

### 1.1 별도 `/ask` 페이지 (Karpathy Simplicity First)

사용자 결정 — `/search` 통합 vs 별도 페이지 → **별도 페이지** (a 옵션).

**Surface Assumption**:
- "UI 통합" 의미 모호 → 사용자 명시적 결정 받음
- `/search` 통합 시 quota trade-off (자동 호출 vs 버튼) + 회귀 위험

**선택 이유**:
- 회귀 0 — 기존 `/search` server component 변경 0
- PoC 검증 최소 — 변경 폭 작음
- quota 보호 — 사용자가 명시적 진입 시에만 LLM 호출

### 1.2 Server Component 단일 fetch (AGENTS.md §1)

```tsx
export default async function AskPage({ searchParams }) {
  const { q, doc_id, top_k } = await searchParams;
  if (!query) redirect('/');
  const response = await getAnswer(query, topK, docId);
  return <main>...</main>;
}
```

- SSR HTML 즉시 표시 (loading.tsx 가 그 사이 skeleton)
- interactivity 불필요 (PoC v1) → `'use client'` 없음
- 검색 페이지의 RSC 패턴 동일 일관성

### 1.3 PoC v1 의도된 minimal 설계

**포함**:
- 답변 텍스트 (whitespace-pre-line)
- 출처 카드 N개 — `[1] doc_title (p.X) · section_title` + chunk_idx + snippet 200자
- 검색 0건 시 "찾지 못함" + 검색 페이지 link
- 빈 query → redirect to `/`
- 503 / generic 에러 분류 (search/error.tsx 패턴 재사용)

**미포함 (v2)**:
- 인용 [N] 클릭 → 출처 카드 하이라이트 / 스크롤
- streaming 답변 (현재 동기)
- `/search` 페이지 답변 영역 통합
- 답변 재생성 / 모델 선택


## 5. Karpathy 가이드라인 적용 회고

**1. Think Before Coding** — "UI 통합" 의미 모호 → Surface assumption + 사용자 명시 결정. 별도 페이지 vs 검색 통합 trade-off 회피.

**2. Simplicity First** — 인용 인터랙션 / streaming / 통합 모두 v2 로 분리. PoC v1 = SSR 답변 + 출처 카드 N개. 변경 5 파일, 신규 의존성 0.

**3. Surface Assumptions** — page 응답이 null vs 0 차이 발견 → frontend 가드 (`page > 0`) 즉시 추가. 백엔드 page null 처리는 후속.

**4. Verifiable Success Criteria** — smoke 3건 (정답 / 자료 없음 / 빈 query) 명시 후 검증. 회귀 0 확인 (tsc + lint).

## 6. 학습

1. **plugin enable 은 settings.json 에 이미 적용됐지만 현 세션에 노출 안 됨** — 새 세션부터 `karpathy-guidelines` 자동 적용. 현 sprint 는 수동 적용 (가이드라인 4 항목 명시).

2. **답변 PoC + 별도 페이지의 회귀 안전성** — `/search` server component 변경 0 으로 기존 검색 흐름 영향 없음. PoC 의 의도된 minimal scope.

3. **Server Component 단일 fetch + Skeleton loading** — 검색 페이지 패턴 동일 일관성. interactivity 필요 없을 때 `'use client'` 회피.

4. **frontend 가드 vs backend 정확화 trade-off** — page=0 graceful frontend 가드 (1줄) vs 백엔드 page null 처리 정확화 (인제스트 파이프라인 변경). PoC 단계는 frontend 가드 우선.
