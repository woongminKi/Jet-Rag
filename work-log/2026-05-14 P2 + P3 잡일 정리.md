## 2. P3 — admin 테스트 2건의 timezone 경계 fragility

### 증상
- `tests/test_admin_feedback.py::AdminFeedbackStatsHappyPathTest::test_basic_mapping` —
  `daily[-1].down = 0` (기대 2)
- `tests/test_admin_queries.py::AdminQueriesStatsHappyPathTest::test_basic_mapping` —
  `daily[-1].count = 1` (기대 2)
- 본 fix 와 무관 (P1 시점에 발견). main HEAD `a7c3a92` 에서도 같은 실패.

### 진단 (자기 비판 3회)

#### 1차: 코드 변경 영향?
- main HEAD 에서도 동일 실패 → P1/P2 변경과 무관.

#### 2차: 무엇이 변했나?
- 본 세션 시작 시점에 시스템 메시지로 날짜 변경 (2026-05-13 → 2026-05-14) 통지.
- 한국 시각이 자정 직후 (KST 00:00~00:29) 가능성 강함.

#### 3차: 코드 로직 분석
- `app/routers/admin.py:227` — `today_kst = datetime.now(KST).date()` (KST 기준 today).
- `app/routers/admin.py:222` 의 `_build_daily_buckets` — row 의 `recorded_at` 을
  `_parse_recorded_at_kst` 로 KST 변환 후 `.date()` 로 bucket 매핑.
- 테스트 row 시각 = `datetime.now(timezone.utc)` 기반 상대 시각.

**KST 00:00~00:29 환경 시뮬 (queries 테스트 기준)**:
- KST 00:15 = UTC 15:15 (어제)
- now_utc = 2026-05-13 15:15 UTC → KST 2026-05-14 00:15 (today)
- now_utc − 30min = 2026-05-13 14:45 UTC → KST 2026-05-13 23:45 (**yesterday**)
- daily[-1] (today, KST) = 1건 (now_utc), daily[-2] (yesterday) = 1건
- 기대 daily[-1].count=2 미달 (1)

→ **확정**: KST 자정 직후 환경에서 row 시각이 KST 자정 경계를 넘어 today/yesterday bucket
분리. 테스트 자체의 timezone fragility.

### Fix
- row 시각을 **KST today 정오 (UTC 03:00) 기준**으로 변경:
  ```python
  base_kst_noon = datetime.now(KST).replace(
      hour=12, minute=0, second=0, microsecond=0,
  )
  base_utc = base_kst_noon.astimezone(timezone.utc)
  ```
- 모든 row (base_utc, base_utc − 10min, base_utc − 20min, base_utc − 30min) 가 KST 정오 근처
  → timezone 경계 무관 same day 보장.
