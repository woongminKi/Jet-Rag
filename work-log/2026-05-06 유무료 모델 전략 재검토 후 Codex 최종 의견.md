## 0. 한 줄 결론

> Claude Code 의 통합안은 큰 방향이 맞다. 다만 그대로 sprint 진입하기 전에 **Multi-query default on**, **Vision 5~10p 고정 cap**, **검색 개선 우선순위**는 수정하는 것이 좋다.

Codex 의 수정된 최종 권고는 다음이다.

1. **Multi-query default on 이 아니라, cheap router default on + paid decomposition gated**
2. **Vision 은 페이지 수 고정 cap 이 아니라, vision_need_score + 문서당 비용 budget 기반**
3. **검색 기능 추가 전에 S0 비용 계측/캐시/cap 을 먼저 완료**

## 1. Claude Code 통합안에 대한 총평

Claude Code 문서는 공정하게 정리되어 있다.

특히 다음 방향은 그대로 채택해도 좋다.

- `retrieval-first`
- Vision 은 선별 호출
- `/answer` 는 사용자 액션 중심
- 유료 모델은 전체 답변보다 질문 분해/시각 정보 보강에 제한 사용
- usage/cost 추적 먼저
- page hash 기반 cache 필요
- reranker candidate cap 필요
- 골든셋은 `query_type`, `negative`, `acceptable_chunks` 를 포함해야 함

다만 Claude Code 문서의 최종 통합안 중 일부는 아직 과감하거나, 구현 순서가 빠르다.

가장 조정해야 할 지점은 다음 3개다.

| 항목 | Claude Code 통합안 | Codex 최종 수정 의견 |
|---|---|---|
| Multi-query | Flash-Lite default on | **router default on + paid decomposition gated** |
| Vision cap | 기본 5~10 pages/doc | **vision_need_score + 비용 budget + page cap 병행** |
| Sprint 순서 | S0 후 골든셋/검색 stack | **S0 비용 계측 없이는 검색 stack 확장 금지** |

## 2. Multi-query: default on 보다 gated 가 낫다

### 2.1 기존 Codex 1차 의견 수정

Codex 1차 의견에서는 query decomposition 을 검색 성능 개선의 1순위로 강하게 권고했다.

그 방향은 유지한다. 하지만 **모든 질문에 paid decomposition 을 기본 적용하는 것은 과하다.**

이유는 비용보다 **검색 노이즈** 때문이다.

Flash-Lite 호출 비용이 작더라도, 잘못 분해된 query 는 검색 후보를 오염시킬 수 있다. 특히 한국어 질문은 생략, 지시어, 맥락 의존 표현이 많아서 LLM 이 없는 맥락을 보태거나 질문 의도를 과잉 해석할 가능성이 있다.

따라서 최종 권고는 다음이다.

```text
모든 질문
→ 무료/룰 기반 cheap router 는 항상 실행
→ 단순 질문이면 단일 검색
→ 복잡/비교/cross-doc/low-confidence 일 때만 Flash-Lite decomposition
→ 그래도 실패하면 HyDE 또는 answer LLM
```

### 2.2 추천 trigger

paid decomposition 은 다음 조건 중 하나 이상이 강할 때만 실행한다.

| trigger | 예시 |
|---|---|
| cross-doc 신호 | "A 문서랑 B 문서 비교", "두 자료에서" |
| 비교/차이 | "차이", "비교", "다른 점", "공통점" |
| 인과/관계 | "왜", "어떻게 연결", "영향", "관계" |
| 변경점 | "바뀐 점", "개정", "전후", "지난번 대비" |
| 긴 질문 | 25자 이상 + 접속어/조건절 포함 |
| 결과 confidence 낮음 | 단일 검색 top score 낮음 또는 top docs 분산 |
| 복수 대상 언급 | 문서명/기관명/날짜/태그가 2개 이상 |

### 2.3 최종 pipeline

```text
사용자 질문
→ NFC 정규화
→ cheap router (룰 기반 + regex + lightweight score)
→ simple intent:
    dense + sparse + doc_embedding 단일 검색
→ complex intent:
    Flash-Lite query decomposition
    sub-query 별 dense + sparse + doc_embedding 검색
→ RRF merge
→ BGE reranker candidate cap
→ MMR/diversity
→ chunk/page/section/table caption 표시
→ 사용자가 원할 때만 /answer 생성
```

### 2.4 결론

`Multi-query default on` 은 `paid decomposition default on` 으로 해석하면 과하다.

정확한 결론은:

> **cheap router 는 default on, paid query decomposition 은 gated on.**

## 3. `/answer`: 사용자 액션 중심, 단 짧은 extractive summary 는 자동 가능

Claude Code 통합안의 `/answer` 사용자 액션 only 전환은 대체로 맞다.

Jet-Rag 의 핵심은 사용자가 문서를 업로드해두고 나중에 빠르게 근거를 찾는 것이다. 따라서 매 검색마다 유료 LLM 답변을 자동 생성하면 비용과 latency 가 커진다.

다만 완전히 "chunk 만 보여주기"로 가면 복잡 질문 UX 가 약할 수 있다.

절충안:

| 레벨 | 동작 | 모델 비용 |
|---|---|---|
| 기본 검색 결과 | top chunk, snippet, 출처, 매칭 강도 | 무료 |
| 자동 요약 | top chunks 에서 extractive summary 또는 template summary | 무료/저비용 |
| 자세한 답변 | 사용자가 "답변 생성" 클릭 | paid |
| 복잡 질문 | 검색 confidence 낮거나 cross-doc 일 때 답변 생성 CTA 강조 | paid optional |

즉, 기본 화면은 빠르게 보여주고, LLM 답변은 사용자가 선택하게 한다.

## 4. Vision: 5~10p 고정 cap 보다 budget 기반 선별

### 4.1 문제

기본 5~10 pages/doc 는 시작점으로 괜찮다.

하지만 고정 cap 만으로는 부족하다.

- 5페이지짜리 스캔 문서는 전부 vision 이 필요할 수 있다.
- 100페이지짜리 텍스트 PDF 는 2페이지만 vision 이 필요할 수 있다.
- 표/그림이 뒤쪽에 있으면 "앞 N페이지" 방식으로 놓친다.
- 중요한 문서와 일반 문서의 비용 허용치가 다르다.

### 4.2 최종 권고

Vision 정책은 다음 3개를 함께 사용한다.

1. **vision_need_score**
2. **page cap**
3. **cost budget**

```text
PDF 전체 무료 분석
→ 각 페이지 vision_need_score 계산
→ score 높은 페이지 정렬
→ page cap 과 cost budget 중 먼저 닿는 지점까지 vision 호출
→ page_hash + prompt_version cache 저장
```

### 4.3 초기 운영값 제안

| 모드 | page cap | cost budget | 동작 |
|---|---:|---:|---|
| 기본 | 10 pages/doc | 약 500원/doc | 일반 사용자 기본값 |
| 품질 우선 | 30 pages/doc | 약 1,500원/doc | 사용자가 중요 문서로 선택 |
| 전체 고품질 | manual only | 실행 전 확인 | 데모/평가/특수 문서 |

금액은 실제 `usage_metadata` 저장 후 조정해야 한다. 지금 단계에서는 절대값보다 **budget 메커니즘** 자체가 중요하다.

### 4.4 vision_need_score 후보 신호

| 신호 | 우선도 |
|---|---|
| PyMuPDF text density 낮음 | 높음 |
| image area ratio 높음 | 높음 |
| table-like block 존재 | 높음 |
| 텍스트 추출 순서 품질 낮음 | 중간 |
| figure/table caption 존재 | 중간 |
| 숫자/단위/기관명 밀도 높음 | 낮음 |
| 페이지가 목차/표지/footer 중심 | 감점 |

## 5. 비용 문제는 검색 개선보다 먼저 계측해야 한다

지금 가장 위험한 것은 "비용이 왜 나갔는지 모르는 상태"다.

검색 정확도 개선 작업을 바로 쌓으면 유료 호출 지점이 더 늘어난다. 그러면 비용 문제의 원인이 더 흐려진다.

따라서 S0 는 반드시 비용 계측/캐시/cap 이다.

### 5.1 S0 필수 항목

| 항목 | 설명 |
|---|---|
| Gemini `usage_metadata` 저장 | input/output/thinking token 과 모델 기록 |
| retry/sweep 실제 호출 수 저장 | 예상 호출 수와 실제 호출 수 차이 확인 |
| page hash cache | 동일 페이지 재호출 금지 |
| prompt_version cache key | prompt 변경 시 cache invalidate |
| estimated_cost 계산 | doc/page/source_type 별 비용 확인 |
| 문서당/일별/user별 cap | 개발 중 비용 폭주 방지 |
| vision_need_score 로그 | 선별 정책 튜닝 근거 확보 |

### 5.2 cap 도달 시 동작

cap 도달 시 인제스트 전체를 실패시키면 안 된다.

권장 동작:

```text
cap 도달
→ 무료 파서 결과는 정상 저장
→ vision 보강만 partial skip
→ documents.flags.vision_budget_exceeded = true
→ UI 에 "시각 보강 일부 생략" 표시
→ 사용자가 품질 우선 모드로 재처리 가능
```

## 6. 인제스트 보조 인덱스 우선순위

Codex 1차 의견의 7개 보조 인덱스는 모두 가치가 있다.

하지만 한 번에 구현하면 범위가 커진다. ROI 순서로 자르는 것이 좋다.

| 우선순위 | 항목 | 이유 |
|---|---|---|
| P1 | 표 caption | 표 검색 실패를 직접 줄임. vision 호출과 같이 추출 가능 |
| P1 | 그림/다이어그램 caption | 현재 PyMuPDF 가 못 보는 정보 보강 |
| P1 | 핵심 엔티티 | 제품명, 기관명, 날짜, 금액 검색에 강함 |
| P2 | section 요약 | 긴 문서 검색 품질 개선 |
| P2 | 동의어 후보 | synonym mismatch 완화 |
| P3 | 예상 질문 | 검색 보조보다 골든셋 draft 에 가까움 |
| P3 | 문서 3줄 요약 고도화 | 이미 일부 있으므로 품질 검증 후 개선 |

예상 질문은 바로 검색 인덱스에 넣기보다, 먼저 골든셋 draft/평가 보조로 쓰는 편이 안전하다. 잘못 생성된 예상 질문을 검색 인덱스에 넣으면 검색 노이즈가 될 수 있다.

## 7. HWP/HWPX vision 누락에 대한 판단

Claude Code 문서가 지적한 HWP/HWPX vision 미적용은 맞다.

Jet-Rag 의 페르소나상 HWP/HWPX 는 중요하다. 하지만 바로 vision 을 붙이기보다 순서를 나눠야 한다.

권장 순서:

1. **한컴 표 추출 품질 개선**
2. HWP/HWPX 내 이미지 binary 추출 가능성 조사
3. 이미지가 실제로 많은 샘플에서만 vision 적용
4. PDF 와 동일한 page/image hash cache 사용

즉, HWP/HWPX vision 은 중요하지만 S0/S1 은 아니다.

우선은 로컬 파서로 뽑을 수 있는 표/텍스트 품질을 높이는 것이 비용 대비 낫다.

## 8. 골든셋과 실 query 로그

골든셋은 검색 로직 개발에 꼭 필요하다.

하지만 자동 골든셋만으로는 실제 사용자의 질문 분포를 충분히 반영하지 못한다.

### 8.1 역할 분리

| 데이터 | 역할 |
|---|---|
| 자동 골든셋 | 실패를 예측하는 테스트셋 |
| 사용자 수정 골든셋 | 실제 페르소나 톤 반영 |
| 실 query 로그 | 실제 실패를 발견하는 데이터 |
| answer feedback | 검색/답변 품질의 운영 신호 |

### 8.2 정답셋 전달 형식

사용자가 Codex 에 줄 때 완벽한 chunk 번호까지 몰라도 된다.

최소한 다음 정도면 평가셋으로 발전시킬 수 있다.

```csv
query,query_type,expected_doc_title,expected_answer_summary,must_include,source_hint,negative
"그때 쏘나타 시트 뭐였지?",fuzzy_memory,"SONATA 카탈로그","트림별 시트 재질 정보","인조가죽;나파가죽","p.22 근처",false
"운영내규랑 직제규정에서 위원회 역할 차이?",cross_doc,"운영내규|직제규정","두 문서의 위원회 역할 비교","위원회;역할;차이","각 문서 위원회 조항",false
"이 자료에 개인정보 보관기간 나와?",out_of_scope,"운영내규","문서에 없음","","",true
```

추후 시스템이 검색 결과를 돌려보고 `relevant_chunks`, `acceptable_chunks` 를 보강하면 된다.

## 9. 수정된 sprint 순서

Claude Code 통합안은 방향이 좋지만, sprint 순서는 조금 바꾸는 편이 낫다.

Codex 최종 권고:

```text
S0 비용 계측/캐시/cap
S1 골든셋 v1 + 실 query 로그 설계
S2 vision_need_score + 선별 vision
S3 gated query decomposition + reranker/MMR 실험
S4 인제스트 보조 인덱스 확장
S5 /answer UX 정리
```

### 9.1 이유

| 순서 | 이유 |
|---|---|
| S0 먼저 | 비용이 왜 나가는지 모르면 어떤 모델 전략도 검증 불가 |
| S1 두 번째 | 검색 개선 전 측정 기준 필요 |
| S2 세 번째 | 현재 비용 hotspot 이 vision 이므로 선별 정책 먼저 |
| S3 네 번째 | 검색 query path 개선은 골든셋과 비용 계측 이후 |
| S4 이후 | 보조 인덱스는 가치 있지만 범위가 큼 |
| S5 마지막 | `/answer` 는 검색 품질과 UX 방향이 어느 정도 잡힌 뒤 정리 |

## 10. Claude Code 에 재검토 요청할 핵심 질문

Claude Code 에 다시 물어볼 때는 다음 질문이 핵심이다.

1. `Multi-query default on` 대신 `cheap router default on + paid decomposition gated` 로 수정하는 것이 더 안전한가?
2. Vision 기본값을 `5~10p 고정 cap` 이 아니라 `vision_need_score + page cap + cost budget` 으로 바꾸는 것이 더 맞는가?
3. S0 비용 계측/캐시/cap 없이는 S1/S2 검색 개선에 들어가지 않는다는 우선순위에 동의하는가?
4. `/answer` 는 사용자 액션 중심으로 두되, 검색 결과 상단에 extractive summary 를 자동 표시하는 절충안이 더 나은가?
5. HWP/HWPX vision 은 중요하지만, 먼저 로컬 표/텍스트 파서 품질을 높인 뒤 적용하는 순서가 맞는가?
6. 예상 질문을 검색 인덱스에 바로 넣지 않고 골든셋 draft 로 먼저 쓰는 것이 더 안전한가?

## 11. 최종 권고

Jet-Rag 의 비용/검색 성능 문제를 동시에 해결하려면 다음 방향이 가장 안전하다.

> **유료 모델을 기본값으로 넓게 깔지 말고, cheap signal 로 필요성을 판정한 뒤 쓰는 계단형 파이프라인으로 만든다.**

최종 구조:

```text
무료 파서/검색/라우터
→ confidence 부족 또는 복잡 질문 감지
→ 저가 유료 모델로 query decomposition
→ 무료 retrieval stack 으로 후보 수집
→ 무료 reranker/MMR 로 정렬
→ 필요할 때만 유료 answer
```

Vision 도 동일하다.

