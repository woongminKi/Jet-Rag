## 1. 게이트 로직 (코드 확인 — W-1(a) 가 `/search` 로 옮길 것)

`/answer` (`api/app/routers/answer.py` L460~492) → `query_decomposer.decompose()` 가 **실제 LLM 호출**하려면 순서대로 전부 통과:
1. **`router_decision.needs_decomposition == True`** — `intent_router._decide_decomposition` 산식 = `(T1 or T2 or T3 or T7) or (T5 and T6)`. (cross_doc P1 의 보조 regex `_T1_CROSS_DOC_PAIR`/`PAIR2`/`PLURAL`·`_DOC_NOUN`·`_T2_COMPARE_STEM` 포함.) — **이게 False 면 즉시 skip, LLM 0.**
2. **ENV `JETRAG_PAID_DECOMPOSITION_ENABLED ∈ {true,1,yes,on}`** — `decompose()` `_is_enabled()`. (본 측정은 ON 가정.)
3. **LRU cache miss** — module-level in-process OrderedDict(200건), key = `(query.lower().strip(), triggered_signals)`. **프로세스 재시작 시 휘발** (마이그 016 의 영구 embed-cache 와 달리 persist 안 함 — eval 재현 시 비용 0 보장 안 됨).
4. **budget cap 미초과** — `vision_usage_log` `source_type='query_decomposition'` SUM > `JETRAG_DECOMPOSITION_MONTHLY_CAP_USD`(default $0.30) 면 skip. DB 부재·실패는 graceful(allowed).
5. **LLM 호출** → JSON parse → **subquery 2~5개여야 채택** (`2 ≤ len ≤ 5` 아니면 빈 tuple = no-op). 1개·6개+·파싱 실패·LLM raise 면 cost 0 (단 LLM raise 시 실 API 비용은 발생, `_record_usage` 안 됨).
- **retry**: `_gemini_common._MAX_ATTEMPTS = int(ENV JETRAG_GEMINI_RETRY, "1")` = **기본 1 (재시도 없음)**. → retry factor ≈ 1.
- ※ `_LOW_CONFIDENCE_THRESHOLD=0.75` (`answer.py` L52) 는 `answer_meta["low_confidence"]` 마킹에만 쓰임. `decompose()` 호출 전 confidence 비교 분기 **없음** (L472 무조건 호출). → 실 게이트 = needs_decomposition + ENV.

## 3. 비용 추정 (PRD §3 W-1(a) "~$0.005/eval" 평가)

`_estimate_cost_usd` (char/4 ≈ token 근사): system 388자 + user `질문: {q}\nJSON:` ≈ 410~450자 → input ~105~113 tok / output(`_LLM_MAX_OUTPUT_TOKENS=200`, 실 3 subquery ~100~200자) ~25~50 tok / gemini-2.5-flash-lite $0.10·$0.40 per 1M → **~$0.000026 / firing** (retry 1).

| 기준 | per-firing | × 20 firing (eval 1회) |
|---|---|---|
| PRD §3 가정 | ~$0.0002 | ~$0.004 (PRD 본문 "~$0.005") |
| 코드 `_estimate_cost_usd` | ~$0.000026 | **~$0.0005** |
| 보수(한국어 token ≈ char/4 ×1.5~2) | ~$0.00004~0.00006 | ~$0.0008~0.0012 |

→ PRD "~$0.005/eval" = **4~10배 보수적 과대** (안전 측, 그대로 두어도 무방). 실 = eval 1회 **$0.0005~0.0012**. "게이트 통과 = LLM 호출" 거의 1:1 (cache cold, budget 여유) — 단 subquery 채택(`2≤len≤5`)은 LLM 품질 의존이라 본 dry-run 으로 예측 불가.

## 4. latency (PRD "전체 ~5%, p95 무영향" 검증)

발화 11% > PRD 가정 5% → **이론상 p95 에 닿을 수 있음** (cross_doc 5% + FP 6% 의 decompose query 가 +1~2초 추정 — sub-query 별 추가 RPC(top_k=10×N_sub) + RRF merge). 실 측정 필요. → W-1(a) 구현 시: ① 응답에 `decomposition_fired`/`decomposed_subqueries`/`decomposition_cost_usd` meta (`/answer` 와 동일 schema) 노출 — eval 에서 latency 분리·비용 집계 ② decompose LLM 호출에 짧은 timeout 가드 (현 `with_retry` 에 timeout 없음, Gemini SDK 기본 의존).

## 5. W-1(a) 진입 권고 / 착수 전 확인 사항

**진입 가능 — 조건부.** 비용 무문제(eval 1회 ~$0.001), cross_doc 9/9·FN 0, FP 6.4% 수용 가능(전부 단일-doc, decompose 가 원본 top_k=20 > sub top_k=10 RRF merge 라 단일-doc 회귀 위험 낮음 — 단 **W-1(a) 본 구현 후 eval 재측정으로 단일-doc 회귀 0 재검증 필수**).

착수 전 확인:
1. **게이트 = needs_decomposition AND ENV 두 개뿐 — confidence 임계 끼워넣지 말 것** (spec 명시). `answer.py` `_LOW_CONFIDENCE_THRESHOLD` 은 meta 전용 dead-ish, docstring 미구현.
2. `/search` 는 이미 `intent_router.route()` 를 2회 호출(`_is_cross_doc_query` MMR / `_is_cross_doc_class_query` chunk-cap). W-1(a) 추가 시 3회 안 되게 **decision 1회 산출 후 재사용** 리팩토링 권고 (룰 기반이라 비용 0 이나 위생).
3. cache 영속성 없음 (in-process LRU 200). eval 재현성 위해 영구화할지 결정 — MVP 범위에서는 in-process 유지 무난 (eval 비용 $0.001 라 영구화 ROI 낮음).
4. subquery 채택률·cross_doc R@10 변동·단일-doc 회귀·latency 분포 = **W-1(a) 본 구현 직후 `JETRAG_PAID_DECOMPOSITION_ENABLED=true` 로 eval 1회(~$0.001)** 돌려 한 번에 측정. 본 게이트 검증은 그 사전 단계로 완료.
5. **P2 함정 재확인** — P2 ($0 기계분해)가 net-negative 였던 이유 = doc-scoped 검색이 진짜 doc-RRF 아님. paid LLM decompose 도 sub-query 별 검색이 여전히 전-doc 풀(`_DECOMP_TOP_K_PER_SUB=10` 글로벌)이라 cross_doc 라벨 doc-bound 매칭에 안 맞을 가능성 — W-1(a) eval 에서 cross_doc R@10 이 실제로 0.4424→상승하는지 가장 먼저 확인. 안 오르면 (b) doc-scoped RPC 마이그로 fallback (DECISION-2).
