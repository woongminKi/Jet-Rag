## 3. 항목별 상세 명세 (P0 / P1 한정)

### 3.A 업로드 SLO 회복 — BackgroundTask 분리 (P0, 0.5d)

**Why**
- 페르소나 A Primary Golden Path (월요일 08:50, §6.1) 는 업로드 → 검색 10분 내에 닫혀야 한다. 8.6MB 3.7초는 **업로드 응답 체감 지연** 으로 "접수됨 < 2초" (§10.11) 기대 깨짐
- 이후 W2 의 HWPX (수십 MB HWPX 흔함) · 이미지 많은 PDF (§10.11 "20+ 이미지 3분") 작업이 이 회복 위에서 측정되어야 함

**What**
- `POST /documents` 요청 흐름을 **"Storage 업로드 → 즉시 job 생성 → 202 Accepted 응답"** 로 재정렬
- 현재: `Storage.upload → documents insert → BackgroundTask(파이프라인)` → 업로드 자체가 동기
- 변경: `multipart 수신 → 바이트 버퍼 → BackgroundTask(Storage.upload + pipeline)` → 수신 즉시 응답
- 응답 스키마 불변 (`document_id`, `job_id`, `status`) — 프론트 코드 무영향
- 실패 핸들링: Storage 업로드 실패도 BackgroundTask 안에서 일어나므로, `documents.flags.upload_failed` 플래그 신설 + 기존 failed 정책 (W1 Day 7+) 과 동일 경로로 통합

**엣지 케이스**
- 수신 중 연결 끊김 → BackgroundTask 가 orphan 업로드를 만들지 않도록 `request.is_disconnected()` 감지 또는 Storage 업로드 전 SHA-256 재계산으로 무결성 확인
- 50MB 상한 (§11.3 단계 A) 은 **수신 단계에서 스트리밍 카운터로 차단** — 바이트 버퍼에 다 담은 뒤 거부하면 메모리 낭비
- 동일 파일 빠른 연속 업로드 → job 2개 생성될 수 있음. dedup 스테이지에서 SHA-256 충돌 시 후행 job 을 "중복으로 skip" 상태로 완결 처리

**수용 기준 (AC)**
- [ ] 50MB PDF 업로드 수신 응답 P95 < 2.0초 (10회 연속 측정)
- [ ] 8.6MB (W1 실측 파일 그대로) 수신 응답 < 1.5초
- [ ] 업로드 중 Ctrl+C (연결 끊김) 시 orphan document row 미생성
- [ ] 기존 프론트 UploadItem 흐름 무변경 (폴링·재시도 포함)

### 3.B Vision 경로 — `VisionCaptioner` 인터페이스 + Gemini 구현 (P0, 1.5d)

**Why**
- T2 는 §5 에서 MVP 필수 판정. 현재 어댑터 5종 중 Vision 은 빠져있음 (§9.4 의 `parser.py` 에 Image 만 명시, 캡셔너 분리 미정)
- US-03 (표·이미지 역검색) + US-07 (화이트보드 액션 아이템) 양쪽 차단 해제
- 기획서 §10.4 의 **단일 호출 4필드 JSON 프롬프트** 가 Vision 경로의 표준 계약 — 이를 코드로 박는다

**What**
- 신규 Protocol: `VisionCaptioner.caption(image_bytes: bytes, mime: str) -> VisionCaption`
- 반환 타입 `VisionCaption` (§10.4 스키마 1:1 대응):
  ```
  type: Literal["문서","스크린샷","메신저대화","화이트보드","명함","차트","표","기타"]
  ocr_text: str
  caption: str
  structured: dict | None
  raw_confidence: float | None  # Gemini response 에 있으면
  ```
- 구현: `GeminiVisionCaptioner` (기존 `GeminiLLMProvider` 와 **어댑터 분리**, §9.4 재구성)
- 이미지 다운스케일 정책: **max 1024px 단변** (§15.2 DE-06 기본 제안) → W2 Day 2 실측 후 **DE-06 확정**
- 원본은 Storage 에 그대로 보관, Vision 에는 다운스케일본 전송 (§9.6 "원본 Storage 전제")

**어댑터 레이어 영향 (§9.4 수정 필요)**
```
app/adapters/
├── vision.py       # [신규] VisionCaptioner: Gemini(기본) / OpenAI(스텁)
├── llm.py          # 텍스트 생성 전용으로 범위 축소
└── parser.py       # ImageParser 는 VisionCaptioner 를 composition
```
→ 기획서 §9.4 를 **v0.2 에서 갱신** 해야 함 (남은 결정 DE-14 참조).

**엣지 케이스**
- HEIC (iPhone 기본) → `pillow-heif` 변환 선행 (§10.3). **사전 승인 필요** — 의존성 추가
- 회전 EXIF — Gemini 가 자체적으로 처리하는지 불확실, 실측 필요. 실패 시 `PIL.ImageOps.exif_transpose` 선처리
- 한국어 OCR 환각 (숫자·한자) → §12.3 "2등 시민" 원칙대로 `confidence_tier` 메타로 기록, 답변 UX 는 W4
- 이미지 내 이미지 (메신저 스샷 안의 사진) → 1-pass 로 제한 (v2 에서 2-pass)
- 파일 크기 2MB+ 원본 → 다운스케일 후에도 Gemini 한도 초과 가능. 초과 시 512px 재시도
- JSON 파싱 실패 → 3회 retry 후 `ocr_text` 빈 문자열 + `flags.vision_partial`

**수용 기준**
- [ ] 메신저 스샷 10건 중 9건 `type == "메신저대화"` 정확 분류
- [ ] 문서 스샷 10건 중 8건 OCR text 가 원문과 문자 일치도 90%+ (수동 검수)
- [ ] 화이트보드 3건 — OCR 부정확해도 `caption` 에 요지 포함 (§12.5 "v2 이관" 이지만 프롬프트는 동작해야 함)
- [ ] 단일 이미지 처리 P95 < 15초 (§10.11)

### 3.C HWPX 파서 (P0, 1.5d)

**Why**

- 포트폴리오 차별점 1번 (§8.3) — "HWP 까지 읽는 한국어 RAG" 의 물증

**What**
- 신규: `HwpxParser(DocumentParser)` — `python-hwpx` 사용 (라이브러리 기존, **의존성 추가 사전 승인 필요**)
- 추출 결과: PyMuPDF 와 동일한 `ParsedDocument` 형태 (text + page + bbox 가능한 범위) — 어댑터 대칭성 유지
- HWPX 내부 이미지 → Vision 경로로 재라우팅 (B 의존)
- HWPX 내부 표 → markdown 변환 (§10.4 `structured` 스키마 재사용)
- `DocumentParser.match(mime, extension)` 디스패처 확장: `.hwpx` → `HwpxParser`

**엣지 케이스**
- **한국어 조사·띄어쓰기**: HWPX XML 은 space 처리가 일관되지 않음 — 추출 후 `re.sub(r'\s+', ' ', text)` 정규화 + 청킹 전 한국어 문장 경계 추가 검증
- **한자 병기**: `한국어(韓國語)` 패턴 보존 (청킹·검색 양쪽에서 의미 손실 방지)
- **섹션 헤딩 메타**: §10.5 1차 분할 기준. HWPX `<hp:head>` 스타일 추출
- **표 안의 표** (중첩): 외곽만 추출, 내부는 플레인 텍스트로 폴백
- **암호화 HWPX**: §11.3 단계 A 에서 거부되어야 함 — 파서가 명시적으로 `HwpxEncryptedError` raise
- **빈 문서 / 표지만 있는 문서**: 200자 미만 → §10.5 3차 병합 로직이 처리. 추가 작업 없음
- **구 HWPX** (2010 초기 버전): 스키마 차이 발생 시 hwp5html CLI 로 폴백

**수용 기준**
- [ ] 합성 HWPX 10건 (섹션·표·이미지·각주·목차 혼재) 중 9건 chunk 추출 성공 (§13.1 HWP 성공률 기준선)
- [ ] 한국어 조사·띄어쓰기·한자 병기 보존 — 수동 3건 검수
- [ ] HWPX 50p 처리 P95 < 90초 (§10.11 HWP 구포맷 SLO 준용)

### 3.D 이미지 파서 (P0, 1.0d)

**Why**
- 수신 경로 5종 중 **클립보드 · 카메라** 가 이미지 직결 (§9.8)
- Secondary / Tertiary Golden Path (§6.2, §6.3) 는 모두 이미지 자료 수신이 빈번

**What**
- 신규: `ImageParser(DocumentParser)` — `VisionCaptioner` 를 composition 으로 사용
- 처리 흐름: HEIC 변환 → EXIF 정규화 → 다운스케일 → Vision 호출 → `ocr_text` + `caption` 을 각각 chunk 로 등록
- chunk 구조: **1 이미지 = 2 chunks**
  - chunk 0: `ocr_text` (검색용, BM25 우세)
  - chunk 1: `caption` (의미 검색용, Vector 우세)
- 메타: `source_channel="image"`, `confidence_tier=Vision의 type 에서 매핑` (§12.3)

**엣지 케이스**
- **PNG·JPEG·HEIC·WebP** 외 포맷 — 단계 A 에서 거부
- **GIF 애니메이션** — 첫 프레임만 추출 (PIL)
- **회전** — EXIF orientation 처리 필수 (B 명세와 동일)
- **초고해상도** (8K 스샷 등) — 메모리 보호를 위해 수신 단계에서 8192px 초과 시 거부
- **동일 이미지 여러 번** (카톡에서 같은 스샷 전달) — SHA-256 dedup (Tier 1) 이 자동 차단
- **투명 배경 PNG** — 다운스케일 시 배경 흰색으로 flatten (Vision 호환성)
- **EXIF GPS** — §15.3 DE-11 "GPS 제외" 기본 제안 준수. 촬영시각만 `created_at_source` 필드로 저장 후 EXIF 폐기

**수용 기준**
- [ ] 5종 수신 경로 중 클립보드 붙여넣기 업로드 → chunk 2개 생성 확인 (수동)
- [ ] HEIC 3건 업로드 → 변환 성공 + Vision 호출 성공
- [ ] 회전된 이미지 (iPhone 세로 촬영) → OCR 정상
- [ ] GPS EXIF 가 저장 경로에 남지 않음 (§15.3 DE-11)

### 3.E URL 파서 (P1, 1.0d)

**Why**
- Tertiary Golden Path (§6.3) — "팀 슬랙의 외부 레퍼런스 PDF → iOS Shortcuts" 의 변종으로 **URL 직접 공유** 가 아이폰에서 가장 자연스러운 경로
- 스샷보다 URL 이 품질 우수 (§12.1 "웹페이지 스샷 → URL 수집이 더 우수")

**What**
- 신규: `UrlParser(DocumentParser)` — `trafilatura` 사용 (의존성 추가 **사전 승인 필요**)
- 입력: `POST /documents` 에 `url` 필드 추가 (파일 대신) — multipart 양립 가능
- 흐름: fetch → trafilatura extract → `title`, `author`, `published_date`, `body_markdown` → 청킹
- `source_channel="url"`, `source_url` 메타 기록

**엣지 케이스**
- **로그인 벽 페이지** — 내용이 "로그인 해주세요" 로 들어옴. 추출된 본문 200자 미만 + "로그인" 키워드 포함 시 `flags.auth_wall=true` + 사용자 경고
- **JavaScript 렌더링 SPA** — trafilatura 가 빈 body 반환. `flags.empty_body=true` 로 처리 후 사용자에게 "동적 페이지라 추출 제한" 안내
- **Paywall** — 일부만 추출됨. 본문 길이 + 키워드 `"구독"`, `"프리미엄"` 으로 휴리스틱 감지
- **PDF URL** — Content-Type 검사로 `application/pdf` 감지 시 다운로드 후 PDF 파서로 재라우팅 (동일 doc_id 유지)
- **한국어 인코딩 오탐** (EUC-KR 잔존) — `chardet` 재검출 (trafilatura 내장 기능)
- **robots.txt 차단 사이트** — trafilatura 는 무시하지만, 포트폴리오 맥락상 **User-Agent 명시** (`Jet-Rag/0.1 (+github_repo_url)`) + 실패 시 거부 안내
- **SSRF 방어** — localhost · 사설 IP 대역 차단 (DE-14 신규 결정 필요)

**수용 기준**
- [ ] 공개 뉴스 URL 5건 → 본문 추출 + chunk 생성 성공
- [ ] 로그인 벽 사이트 1건 → `auth_wall` 플래그 + UI 경고
- [ ] PDF URL 1건 → PDF 파서 경로 진입
- [ ] localhost/192.168.x.x URL → 거부 (SSRF)

### 3.F HWP 구포맷 (P1, 1.0d)

**Why**
- 공공기관 자료는 여전히 구 HWP 5.x 비율 높음 (§2.1). 완전 제외 시 "HWP 까지 읽는" 브랜드 메시지 약화

**What**
- 신규: `Hwp5CliParser(DocumentParser)` — `hwp5html` 외부 CLI 서브프로세스 호출 (§9.2)
- 시스템 의존성: `hwp5html` 설치 필요. README 업데이트 + 배포 환경 (Railway) 에서의 설치 가능성 **W2 Day 1 선행 조사**
- graceful degradation (§10.3): 실패 시 `HwpLegacyUnsupportedError` raise + 사용자에게 "HWPX 재저장 권장" 메시지

**엣지 케이스**
- **Railway 환경에 hwp5html 설치 불가** → W2 범위에서 제외, DE-13 기본 제안 ("10% 허용치 / 이상 시 HWPX 1등만") 을 **HWP 구포맷 전면 제외로 격상** 결정
- **암호 걸린 HWP** → 거부
- **CLI timeout** — 60초 hard limit, subprocess.kill
- **출력 HTML 파싱** — BeautifulSoup 기반, 표·이미지 참조 추출
- **인코딩** — hwp5html 은 UTF-8 출력. 만약 깨지면 재시도 없이 실패 처리

**수용 기준**
- [ ] 합성 HWP 5건 중 3건 chunk 추출 성공 (§13.1 성공률 기준 완화 — DE-13 허용치 참조)
- [ ] 실패 2건 → 사용자에게 "HWPX 재저장" 안내 메시지 노출
- [ ] Railway 배포 환경 설치 가능 여부 Day 1 에 결론

### 3.G 입력 게이트 B — 콘텐츠 수준 (P1, 1.0d)

**Why**
- §11.5 포지셔닝 유지 장치의 **기술적 구현부**. 경고 UX 카테고리 3종 (§9.7) 을 실제로 작동시키는 첫 지점
- W2 에서 이미지·HWPX 가 들어오기 시작하므로 **게이트 B 가 없으면 "업무 DB" 로 오해될 경로** 가 즉시 열림

**What**
- 파이프라인에 **단계 6 "content_gate"** 삽입 (현재 7-stage → 8-stage)
  - 위치: `extract → chunk → **content_gate** → tag_summarize → ...`
  - 이유: 청킹된 텍스트 전체를 대상으로 스캔 필요
- 검출 로직:
  - PII 정규식 (§11.3): 주민번호 · 카드번호 · 계좌번호 — 발견 시 `flags.has_pii=true` + 위치 (chunk_idx, char_range) 메타
  - 워터마크 키워드: `대외비 | CONFIDENTIAL | 내부자료 | 보안` — 대소문자 무시
  - 메신저 대화: Vision 결과 `type == "메신저대화"` 참조 (이미지만) — `flags.third_party=true`
- **저장은 통과**, UI 에는 §9.7 카테고리별 경고 — 경고 표시는 W4 (S3 문서 상세) 에서 소비
- 사용자 차단 없이 **메타만 기록** — 기획서 §11.3 "결과: 경고(저장)" 준수

**엣지 케이스**
- **주민번호 오탐**: `010-0000-0000` 전화번호 ≠ 주민번호 — 앞자리 6자리 패턴 엄격화
- **카드번호 오탐**: 거래 내역·일련번호 가 맞을 수 있음 — Luhn 체크 추가 고려 (W3 이월 가능)
- **`대외비` 가 본문 인용**: "대외비 해제된 자료입니다" 같은 문맥 → 게이트 B 는 단순 매칭, 오탐은 UX 에서 `[경고 끄기]` 토글로 수용 (§9.7 카테고리별 독립 토글)
- **한자 병기 주민번호** (가상): 패턴 매칭은 ASCII 숫자만
- **Vision `type` 이 `"기타"` 로 회색지대**: 메신저 대화인지 애매할 때 → 게이트 발동 안 함, 사용자가 직접 태그로 분류

**수용 기준**
- [ ] 주민번호 포함 테스트 문서 5건 → 전부 `has_pii=true`
- [ ] `대외비` 워터마크 문서 3건 → 전부 `has_watermark=true`
- [ ] 메신저 스샷 5건 → 전부 `third_party=true`
- [ ] 정상 문서 10건 중 0건 오탐 (false positive rate 0%)

### 3.H 동시 업로드 폴러 최적화 (P1, 0.5d)

**Why**
- W1 §6 명시 이월. 현재 업로드 탭 N개 = 폴링 요청 N개 병렬 (1.5s 간격) → 5개 동시 업로드 시 초당 3.3 req/s 의 status 폴링
- W2 에서 이미지·URL·HWPX 동시 업로드 시나리오 (Secondary 지하철 상황) 에서 UX 끊김 유발

**What**
- 신규: `GET /documents/status?ids=<id1>,<id2>,...` batch 엔드포인트
- 프론트 `useJobStatusPolling` → `useBatchJobStatusPolling` 으로 승격
  - 활성 UploadItem 의 `job_id` 집합을 모아 1개 폴링 요청으로 처리
  - 모두 완료/실패 시 폴러 중단 (기존 로직 유지)
- 응답 스키마: `{ jobs: JobStatus[] }` — 기존 단건 스키마를 배열화

**엣지 케이스**
- **동시 업로드 20개 초과** — URL 길이 제한 걸림. batch 최대 10건 초과 시 2회로 분할
- **하나만 실패** — 배열 안에서 개별 `status="failed"` 유지, 폴러는 나머지 이어감
- **URL 쿼리 인코딩** — 콤마 대신 `ids[]=` 반복으로 전환 가능하지만 GET 유지를 위해 콤마 유지

**수용 기준**
- [ ] 5개 동시 업로드 시 API 호출 수가 W1 대비 1/5 로 감소 (네트워크 탭 실측)
- [ ] 단건 폴러도 batch 엔드포인트 사용 (코드 일원화)
- [ ] 실패 1건 + 완료 4건 혼합 시 UI 각각 정상 표시
