### 2.3 해석

- **청크 수 폭증 (×2.3)** — 4.6 (표 셀 격리) + 4.3 (따옴표 보호) + 4.4 (overlap) 가 실제 파서 출력 (long sections + short cells) 에 적용. dry-run 의 "기존 chunks → resplit" 시나리오와 다른 결과 (실 파서 시작점이 더 fragmented).
- **avg/p50 길이 감소** — 표 셀 + 헤더/푸터 청크가 짧아 (`p50=26`) 평균 끌어내림. 의미 있는 본문 청크는 여전히 적정 길이 유지.
- **table_noise 67.4% 마킹** — 표 셀 분리 효과로 자동 마킹 활성. 검색에서 자동 제외 → 본문 청크 ranking 안정.
- **header_footer 19.2% 신규 마킹** — 짧은 청크 분리로 페이지 반복 헤더가 (b) 룰 임계 (≥3회 + < 100자) 충족. W4-Q-15 (b) 룰의 인프라 사전 준비가 실효 활성.

### 3.2 trade-off 평가

- **top-3 100%** 는 큰 진전 — 모든 정답이 top-3 안에 있음. 사용자가 결과 카드 3장 보는 시나리오 100% 적중.
- **top-1 -5pp** 는 표 셀 격리의 부수효과. 표 안의 숫자/키워드 매칭이 약화됨.
- **KPI §13.1 출처 일치율 ≥ 0.95** — top-1 기준 90%, top-3 기준 100%. 광의 (top-3) 충족, 협의 (top-1) 미달.
- **latency** 미세 증가 (159 → 169ms, +6%) — chunks 폭증 (×2.3) 으로 PostgREST/RPC 부하 증가, embed cache 효과로 흡수.

## 4. 비판적 자가 검토

1. **dry-run vs 실 결과 괴리**: dry-run Δ=0% (정상) → 실 +126%. 사유: dry-run 입력이 기존 chunks (이미 잘 분리된 상태) 였으나 실 reingest 는 파서 raw 출력 (long + short 혼재) 에서 시작. **dryrun_chunk_repolicy.py 의 worst-case 시나리오 (-67%) 도 부정확** — 양극단을 다 빗나감. 향후 dry-run 정확도 향상 필요 (실 파서 결과를 입력으로 시뮬).
2. **chunks 폭증의 비용**: 1256 chunks × 1024-dim float = 5MB+ vector 인덱스. 사용자 자료 누적 (50+ doc) 시 ~50MB. pgvector HNSW 인덱싱 부담 ↑. W6+ 자산 누적 시 모니터링.
3. **top-1 90% 후퇴 회수 옵션**:
   - (A) `_looks_like_table_cell` 임계 강화 (loose → 엄격) — false positive 감소
   - (B) chunk_filter table_noise 임계 완화 (0.9 → 0.85) — 더 많이 마킹하지만 본문 일부도 마킹 risk
   - (C) RRF 가중치 sparse 우세로 변경 — 키워드 hit 강화
   - W6+ 정량 측정 base 확보 후 ablation 권장. 현재는 trade-off 인정 + top-3 100% 충족.
4. **HF API 비용**: 555 → 1256 chunks reingest 시 embed cache miss 가 다수. 실측: ~10분 (cache 일부 hit). 누적 시 cache size (512) 한계 검토 필요.
5. **stuck queued (3970feab)**: BackgroundTasks 의 race? 또는 prior failed job 의 conflict? 향후 reproduction 시 분석. 현재는 manual cleanup 으로 회피.
