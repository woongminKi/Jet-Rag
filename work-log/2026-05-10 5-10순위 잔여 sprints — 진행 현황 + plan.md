## 2. #7 acceptable_chunks LLM-judge 자동 보완 — helper ship ✅

### 2.1 신규 — `evals/_acceptable_judge.py` (~120 LOC)

motivation: 1순위 sprint 의 ILIKE 한계 (R@10 0.37, 키워드 매칭 ≠ 의미 매칭) 후속.

**핵심 API**:
```python
from _acceptable_judge import build_judge_prompt, parse_judgment, select_acceptable

prompt = build_judge_prompt(
    query="한마음 시행일",
    candidates=[(10, "이 내규는 2022년 7월..."), (50, "회원카드 발급...")]
)
# LLM call → raw_json
judgments = parse_judgment(raw_json, expected_indices=[10, 50])
acceptable_idx = select_acceptable(judgments, threshold=0.5, max_count=5)
# → [10] (score >= 0.5)
```

**알고리즘**:
1. query + (chunk_idx, text snippet 300자) JSON 으로 LLM prompt
2. LLM 이 각 chunk 0.0~1.0 점수 + 짧은 reason 반환
3. parse + threshold 0.5 이상 chunk_idx 선택 (+ max_count cap)

**graceful**:
- markdown fence 자동 제거 (LLM 가 ```json ... ``` 응답해도 parse)
- 누락 chunk → score=0 (LLM 미응답 graceful)
- score 자동 clip [0.0, 1.0]

### 2.2 단위 테스트 (11 신규)

- `BuildJudgePromptTest` (2): query/candidates 포함 + 긴 text truncate
- `ParseJudgmentTest` (6): 정상 / fence / 누락 default / clip / invalid raise / non-array raise
- `SelectAcceptableTest` (3): threshold filter / max_count cap / empty graceful

### 2.3 RAGAS regression / golden v2 통합 (별도 sprint)

본 sprint 는 helper module + tests. 실 적용 별도 sprint 필요:
- 178 row 의 acceptable_chunks 보완: ~$1~2.7 (cap 권고)
- candidate 1 row 만 보완 (cost ~$0.005~0.015) sample sprint 로 검증
- 통합 시 cost guard 적용
