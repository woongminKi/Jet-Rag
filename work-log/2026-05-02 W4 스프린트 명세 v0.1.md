### 3.W4-Q-3 — embedding cache (P0, 1.0d)

**Why**: W3 라이브 smoke 결과 cold start 638ms 1건 측정. 페르소나 A 의 모바일 (Android/iOS) 검색 시 매번 cold 면 P95 SLO 위협. 마스터 기획서 KPI §13.1 의 "P95 검색 응답 ≤ 3초" 절대 기준은 충족하나 v0.5 §1.2 의 "W3 < 500ms" 자체 목표 미달.

**What**:
- `app/adapters/impl/bgem3_hf_embedding.py` 의 `embed_query` 캐시 레이어 추가
- 캐시 키: query 문자열 + 모델 버전 (BGE-M3 fixed)
- 캐시 저장소: in-process LRU (Python `functools.lru_cache` 또는 `cachetools.LRUCache`) — 의존성 추가 없이
- 크기: maxsize=512 (페르소나 A 의 일일 쿼리 ~30건 가정, 2주 윈도우)
- TTL: 없음 (LRU만, 모델 버전 바뀌면 codebase 차원 invalidate)
- W3 Day 4 의 `bgem3_singleton` 패턴과 양립 — 싱글톤 안에 LRU 부착

**작업 분해**:
1. `embed_query` 시그니처 무변경, 내부에 LRU 부착
2. 단위 테스트 — 동일 query 두 번째 호출이 < 1ms (네트워크 호출 0)
3. `search_metrics` ring buffer 에 `embed_cache_hit` flag 추가 (가시성)
4. 라이브 smoke — 동일 query 첫 호출 vs 두 번째 호출 took_ms 비교

**DoD**:
- [ ] LRU 캐시 ship + cache hit 시 took_ms < 100ms (warm path 와 동등)
- [ ] 단위 테스트 4건 (hit/miss/eviction/모델 버전 분리)
- [ ] `search_metrics` 에 cache_hit flag 노출
- [ ] golden 20건 batch 실행 시 같은 query 재호출의 took_ms 평균 ≥ 80% 단축

**KPI 매핑**: §13.1 P95 검색 응답 (warm/cold gap 해소).

**테스트 전략**: 단위 (4건) + 라이브 smoke (golden 20건 batch 2회 실행, 첫번째 vs 두번째 평균 비교).

### 3.W4-Q-17 — PDF heading 추출 강화 (P0, 1.0d)

**Why**: chunk 진단 리포트 (2026-04-30) 결과 PDF 4건 모두 section_title 0%. 핸드오프 §3.4 에서 W4 이월 결정. KPI §13.1 의 30% 절대 미달. 코드 cross-check (`pymupdf_parser.py:56`) 가 명시적 `section_title=None` 박아둠. v0.5 §3.F 가 PDF 미커버. 페르소나 A 의 자료 분포 (HWPX 2 + PDF 4 = 6 doc 중 PDF 67%) 에서 가장 큰 갭.

**What**:
- `app/adapters/impl/pymupdf_parser.py` 에 heading 추출 휴리스틱 추가
- 휴리스틱 (청킹 검토 §6 의 4.5 권장과 별개, Day 5 의 HwpxParser graceful degrade 패턴 재사용):

  - **(b) bold 휴리스틱**: `block.spans[*].flags & 16` (PyMuPDF bold flag) → heading 후보
  - **(c) 짧은 라인 + 다음 본문 패턴**: 3줄 이하 + 다음 paragraph 가 일반 본문이면 heading
  - **fallback**: 휴리스틱 모두 실패 시 None (현재와 동일)
- HWPX 의 sticky propagate 패턴 재사용 — heading 발견 시 다음 heading 까지 모든 chunk 의 `section_title` 채움
- `HwpxDocument.open()` graceful degrade 패턴 적용 — PyMuPDF 휴리스틱 fail 시 텍스트 패턴 fallback (chunks 진단 리포트 의 heading_footer 0% 기존 동작 보존)

**작업 분해**:
1. PyMuPDF API 사전 조사 (Explore 에이전트, font size·flags 추출 방법)
2. `_HEADING_FONT_RATIO` (1.15) + `_HEADING_BOLD_FLAG` (16) 상수 정의
3. `_is_heading_block` 함수 + sticky `current_title` 패턴
4. 단위 테스트 — 합성 PDF 1건 (font size 차등) + 사용자 자산 4건 재인제스트
5. 백필 dry-run — 기존 PDF chunks 465건 중 PDF 의 section_title 채움 비율 측정
6. 백필 본 적용 (사용자 confirm 후)

**DoD**:
- [ ] PDF 4건 평균 section_title 채움 비율 ≥ 30% (KPI §13.1 충족)
- [ ] 단위 테스트 (HwpxParser 와 같은 구조, 신규 ~10건)
- [ ] 백필 dry-run + 본 적용 (chunks.flags 와 같은 patterns — doc 별 breakdown)
- [ ] golden 평가셋 PDF 매칭 query 6건 (G-006·007·008·009·010·013·018·019) 의 top-3 chunk 의 section_title 등장 비율 측정 (정성)

**KPI 매핑**: §13.1 section_title ≥ 30% (PDF), 출처 일치율 (heading 정보가 답변 출처 표기에 직접 활용).

**엣지 체크리스트**:
