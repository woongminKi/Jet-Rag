### (A) doc-match-fail row 분모 정책 (DECISION-6)

`run_s4_a_d4_breakdown_eval.py` 가 `recall_at_10 is None` 인 row 를 분모(`n_chunk_evaluable`)에서 전부 제외해 overall R@10 을 inflate 하던 것을 두 버킷으로 분리:

- **`doc_match_fail` 버킷** = `relevant_chunks` 또는 `acceptable_chunks` 가 비어있지 않은데 런타임에서 target items `[]` → `note == "doc 매칭 fail"` 인 row. `_measure_one_cell` 의 `_zero_doc_match_fail()` 헬퍼가 `recall_at_10 / ndcg_at_10 / mrr = 0.0`, `top1_hit = False`, `doc_match_fail_zeroed = True` 로 채워 `n_chunk_evaluable` 분모에 포함.

### (B) dense_vec NULL preflight WARN

`run_s4_a_d4_breakdown_eval.py` `main()` 측정 시작 직전 `_warn_stale_dense_vec(*, strict: bool) -> int`:

- `SELECT id count=exact WHERE dense_vec IS NULL` (Supabase service_role, `_repair_sample_report_dense_vec.py` 패턴 재사용). 0 이면 `[INFO] dense_vec NULL chunk 0건 — OK` 조용히 통과, 0 반환.
- > 0 이면 `[WARN] dense_vec NULL chunk N건 — stale data 가능 (sample-report 2026-05-12 사고 참조)` + doc_id 별 카운트 목록 (`range` 페이지네이션, page size 1000), N 반환.
- Supabase 접속/쿼리 실패 시 `[WARN] dense_vec 점검 skip — Supabase 접속 실패: <reason>` 만 찍고 -1 반환 (eval 막지 않음).
- `--fail-on-null-dense-vec` 플래그 (default False): NULL > 0 시 `main` 이 `return 2`, 측정 중단. 기본 = WARN-only.
