## 3. 주입 메커니즘 (1안 — 마이그 0, text append, strippable 마커)

- **마커 포맷**: chunk text 끝에 `\n\n[검색어: {cand1} {cand2} ...]` (공백 구분) 한 줄. `inject_marker` 결과는 NFC 정규화.
- **cap**: doc-level LLM 후보 쌍 `_CAP_PER_DOC_LLM_PAIRS = 8` / chunk-level 후보 `cap_per_chunk = 5`. dense 임베딩 오염 최소화(마커 ~20~50자).
- **인덱싱 경로**: PGroonga `idx_chunks_text_pgroonga`(`004_pgroonga`, Mecab) 가 `chunks.text` 컬럼만 인덱싱 → 마커가 바로 sparse(`c.text &@~ query_text`) 매칭에 들어감. **마이그레이션 신규 파일 0**. dense(BGE-M3)는 `chunks.text` 임베딩이므로 마커도 함께 임베딩됨(짧은 list → 영향 미미, M2 ablation).
- **snippet 보호**: `search.py` 응답 조립 시 chunk text → snippet 변환 직전에 `strip_synonym_marker` 호출 → 사용자에게 `[검색어:...]` 노출 안 됨. 마커 없는(재인제스트 전) chunk 는 no-op.
- **metadata**: `metadata["synonym_candidates"]`(list, 마커 부착된 chunk 에만), `metadata["synonym_source"]`(`"dict"` 기본 / doc_llm_pairs 가 비어있지 않으면 `"dict+llm"`).
- **idempotency / 위치**: 마커 부착은 2차 split·3차 merge 이후(chunk text 확정 후)에만 1회 → split/merge 로직 무영향. 결과 chunk text 가 `_MAX_SIZE(1000)` 살짝 초과 가능 — DB(`text TEXT NOT NULL`)·임베딩·PGroonga 무제한 허용(코멘트만, 별도 처리 불필요).
- **vision-derived chunk**: `_compose_vision_text` 가 `{base}\n\n[표:...]\n[그림:...]` 합성 후, 그 뒤에 `\n\n[검색어:...]` 가 붙음 → 순서 `[표:...] ... [검색어:...]` (테스트로 가드). 캡션 텍스트에 사전 키가 들어있으면 그 키 기준으로 후보 수집됨.

## 5. 사이드이펙트 점검 결과

- **dense 임베딩(BGE-M3 = `chunks.text`)**: 마커 ~20~50자, doc 8/chunk 5 cap → 영향 미미. ON/OFF dense-only ablation 은 M2 W-4 재인제스트 후 측정 예정(명세에 명시). ENV OFF default → 측정 전까지 영향 0.
- **snippet/하이라이트**: `strip_synonym_marker` 로 마커 제거 후 snippet 생성. ENV OFF / 재인제스트 전 chunk → 마커 없음 → no-op.
- **기존 chunk(재인제스트 전)**: `synonym_candidates` 0건, 마커 없음 — 정상. 마커 있는/없는 chunk 혼재해도 PGroonga·BGE-M3 무문제.
- **chunk_filter / content_gate / dedup**: 마커는 한글 키워드 한 줄 → `table_noise`(short_line_ratio ≥ 0.85 AND digit_punct_ratio ≥ ...) 발화 안 함 — `ChunkFilterRegressionTest.test_marker_appended_body_not_table_noise` 로 확인(`_classify_chunk` → None). content_gate(PII/워터마크)·dedup(doc_embedding) 무관.
- **2차 split / 3차 merge / overlap**: 마커 부착이 split·merge·overlap 후 1회 → 영향 0. 결과 chunk text 가 `_MAX_SIZE(1000)` 살짝 초과 가능 — DB·임베딩·PGroonga 무제한 허용(코멘트만).
- **factory `LLMPurpose` 확장**: 추가만 — 기존 호출처(`tag`/`summary`/`answer`/...) 영향 0. 회귀 가드 `test_factory_pricing_and_master_plan.py` 스냅샷 동기화 완료.
- **LLM (b) RPD**: doc당 1회 무료 1,500/일 안. quota 감지 시 graceful skip(빈 list). default OFF.
- **incremental.py vision path**: `_sections_to_chunks` 에 동일 처리 추가(정적 사전만). ENV OFF 면 무영향.

## 남은 이슈

- **incremental.py 동의어 일관성**: 정적 사전만 적용(doc-level LLM 후보 미생성). incremental 은 vision 보강 chunk 가 캡션 중심이라 LLM 후보 가치가 낮지만, 풀 reingest(`run_pipeline`)와 미세하게 다름 — M2 W-4 풀 클린 재인제스트가 디폴트 경로라 실용상 문제 없음.
- **2안(컬럼 분리 마이그) 전환 트리거**: M2 W-4 ON/OFF ablation 에서 ① dense-only(BGE-M3) R@10 가 −0.01 이상 떨어지거나 ② 단일-doc qtype 회귀가 보이면 → `chunks.search_aug TEXT` 별도 컬럼 + PGroonga 인덱스 마이그(`NNN_*.sql`)로 전환(dense 임베딩은 `chunks.text` 만, sparse 는 `text || search_aug` 매칭). 그 전까진 1안 유지.
- **(b) LLM 후보 풀구현 = M2**: 현재는 최소 구현(프롬프트 1버전, 파싱 graceful, cap 8). M2 에서 프롬프트 정교화 + 비용 cap(`vision_usage_log` source_type 집계)·LRU cache 추가 검토.
- **`metadata.synonym_source` 정밀도**: 현재 `doc_llm_pairs` 가 비어있지 않으면 `"dict+llm"` — 실제로 LLM 후보가 그 chunk 에 매칭됐는지와 무관. 디버그 필드라 허용 범위. 정밀 추적 필요 시 M2 에서 분리.
- **(senior-qa P2 모니터링) 일반어성 사전 키의 cross-doc 오매칭**: `synonym_dict._DOMAIN_SYNONYMS` 의 단어 키 중 비교적 일반어성("동의서"/"회비"/"트림"/"전장" 등)이 여러 문서에 등장 → 마커 주입 시 그 키가 박힌 chunk 가 cross-doc 검색에서 의도치 않게 끌려올 수 있음. 또한 마커 한 줄(`[검색어: ...]`)이 chunk text 끝에 붙으면서 `chunk_filter` 의 `filter_ratio`(table_noise/short_line) 분포가 미세하게 변동 가능. **M2 W-4 전체 클린 재인제스트 + ON/OFF ablation eval 때** 단일-doc qtype 회귀 게이트와 함께 ① cross-doc R@10 변화 ② `filter_ratio` 변동 ③ 일반어 키 오매칭 여부를 함께 확인 — 문제 시 해당 키를 사전에서 제거하거나 2안(`chunks.search_aug` 컬럼 분리)으로 전환.
