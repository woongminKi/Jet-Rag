## 1. 오늘 달성 (Day 6)

### 1.1 백엔드 — `GET /documents/{id}` 단건 조회 엔드포인트
- `app/routers/documents.py` 에 `DocumentDetailResponse` + `get_document(doc_id)` 추가
- `/doc/[id]` 페이지가 한 번에 필요한 정보를 단일 응답으로:
  - title · doc_type · source_channel · size_bytes · content_type
  - tags · summary · flags · created_at · received_ms
  - chunks_count · latest_job (JobStatus)
  - source_url (flags.source_url 추출, URL 인제스트 문서 전용)
- 라우트 충돌 검증: `/documents/batch-status` (static) vs `/documents/{doc_id}` (dynamic) vs `/documents/{doc_id}/status` (다른 segment) — segment 수·정적 부분 차이로 명확 구분
- 404 케이스 처리 — `deleted_at IS NOT NULL` 이거나 doc 없음 시 한국어 detail
- live smoke: 기존 sonata PDF doc → 200 + 모든 필드 정상, 가짜 UUID → 404 정상

### 1.2 프론트 — `/doc/[id]` 페이지 (Client Component)
- `web/src/app/doc/[id]/page.tsx` 신설. Next.js 16 의 `params: Promise<{id: string}>` 패턴 + `use(params)` unwrap
- 폴링 내장: `latest_job.status` 가 queued/running 이면 1.5s 간격 재조회 (max 5분)
- 컴포넌트 구성 (한 파일 안에 함수 분리):
  - **HeroSearch**: 전역 검색 인풋 (DE-27 — 백엔드 변경 0, `/search?q=` 그대로)
  - **DocSummaryHeader**: 제목 + 메타 뱃지 (doc_type · source_channel · 스캔본 · 실패) + 사이즈/날짜/청크/수신 latency + URL 원본 링크
  - **DocStatusSection**: 인제스트 상태 카드 (처리 중 · 완료 · 실패)
  - **SummarySection**: Gemini 요약 (whitespace-pre-line 으로 줄바꿈 보존)
  - **TagsSection**: 태그 chip 들 (각각 `/search?q=tag` 링크)
  - **FlagsSection**: 주의사항 (PII · 워터마크 + 키워드 · 제3자 대화 · 스캔본)
  - **DuplicatedBanner / UploadedBanner**: query string `?duplicated=1` / `?uploaded=1` 시 노출
  - **DocSkeleton / ErrorCard / NotFoundCard**: 로딩/에러/404 폴백

### 1.3 인제스트 자동 이동 (DE-28)
- `app/ingest/page.tsx` 에 `autoRoutedRef = useRef(false)` 추가 — 페이지 lifecycle 동안 1회만 라우팅
- **duplicated=true 즉시 이동**: POST 응답 시 `router.push('/doc/<existing>?duplicated=1')`
- **첫 completed 자동 이동**: `UploadItem` 의 `onCompleted` 콜백 → `router.push('/doc/<id>?uploaded=1')`
- 단일/다중 시나리오 모두 1회만 자동 라우팅 + 이후 completed 들은 "상세 보기" 링크로 처리

### 1.4 UploadItem 확장
- `onCompleted` prop — 인제스트가 completed 로 전이된 시점 1회 호출 (`completedFiredRef` 로 이중 호출 방지)
- "상세 보기" 버튼 — `job.status === 'completed' || data.duplicated` 조건에서 노출 → `/doc/<docId>` 링크
- 기존 useJobStatusPolling/retryNonce 메커니즘은 그대로 유지 (Day 7 polish 까지 보존)

### 1.5 타입·라벨 갱신
- `types.ts`: `StageValue` 에 `content_gate` 추가, `DocType` 에 `url` 추가, `DocumentDetailResponse` + `BatchStatusItem`/`BatchStatusResponse` 신설
- `api/index.ts`: `getDocument(docId)` + `getBatchStatus(docIds[])` 함수
- `doc-type-label.ts`: `url: 'URL'` 라벨 추가
- `stages.ts`: `STAGE_ORDER` 에 `content_gate` 끼워넣고 `STAGE_LABELS`에 "안전 검사" 추가

## 2. 결정

| # | 결정 | 근거 |
|---|---|---|
| **DE-48 (Day 6)** | `/doc/[id]` 페이지는 **Client Component** (Server Component 아님) | 폴링 (queued/running 시 1.5s 재조회) 가 useEffect 기반 — Server Component 로 SSR 후 hydration 으로 폴링 시작하는 패턴은 W4 본격판에서 검토 |
| **DE-49 (Day 6)** | 자동 이동 1회 정책 = 페이지 lifecycle 단위 (`useRef`) | DE-28 의 "다중=첫 완료만 자동" 충실 구현. 페이지 unmount/remount 시 ref 초기화 → 의도된 동작 |
| **DE-50 (Day 6)** | duplicated 즉시 이동 = POST 응답 직후 (BG 대기 X) | dedup 은 SHA-256 매칭 즉시 결정되므로 BG 폴링 불필요. UX latency 최소화 |
| **DE-51 (Day 6)** | Item H 프론트 폴러 batch 전환은 **Day 7 polish** 로 이월 | per-doc 폴링이 1~5건 시나리오에서 충분히 빠름 + retryNonce 메커니즘 보존 우선. backend endpoint 는 Day 5 에 ship 완료 |
