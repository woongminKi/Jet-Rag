## 2. Multi-query 강등 근거 — 비판적 재검토 (4개 모두 약함)

Claude 의 senior-planner 가 Multi-query 를 "C2 비교형 escalation only" 로 강등한 근거 4개를 정직하게 다시 봄.

| # | Claude senior-planner 강등 근거 | 비판적 재검토 (자기방어 인정) |
|---|---|---|
| 1 | "복잡 질문 카테고리 C2 비교형은 ~20% 분포" | **근거 0**. 측정 데이터 없이 senior-planner 가 추정한 수치. 사용자 페르소나 A 의 "다양·복잡 질문 가능성 높음" 명시를 무시한 frame |
| 2 | "latency +0.5~1.5s 추가" | 사실이지만 cache 로 완화 가능. 답변 LLM 1초+ 보다 작음. SLO 3s 안쪽 |
| 3 | "1 query = 5+ paid 호출 누적" | **모델 가정 오류**. Claude 는 Gemini Flash 가정, **Codex 는 Flash-Lite 권고**. Flash-Lite query 분해 = ~$0.0002/회 vs 답변 생성 = ~$0.005/회 → **25배 차이**. 비용 누적 frame 자체가 부정확 |
| 4 | "한국어 query 분해 의도 왜곡 가능" | 정성적·미검증. Codex 가 든 명시 예시 (`A 정책 ↔ B 예산`) 같은 cross-doc 의도엔 분해가 정답에 가까움 |

### 2.1 Codex 가 Multi-query 를 P2 1순위로 둔 근거 정합성

| 근거 | 검증 |
|---|---|
| **사용자 결정 B (기억 보조 페르소나) + Q8 (`/answer` 무료 유지) 와 일치** | 답변 LLM 에 paid 안 쓰기로 했으면 그 자원을 검색 단계 query decomposition 에 쓰는 게 자원 배분상 합리적 |
| **Flash-Lite 비용 무시 가능** | 50q/일 × 30일 × $0.0002 ≈ **$0.30/월**. 운영 위협 0 |
| **페르소나 매칭** | 페르소나 A "다양·복잡 질문" 명시 → cross-doc·비교·인과 비중 ↑ → multi-query 효과 영역과 정확히 일치 |
| **Codex 철학 정합** | "유료 LLM 으로 답변 생성보다 검색 전후 구조 강화가 비용 대비 효과 좋다" — retrieval-first 원칙 1번 도구가 query decomposition |

→ **Codex 권고가 Claude 권고보다 합리적**. Claude 의 escalation only 는 자기방어 후 강등 결정.

## 3. Claude 가 "더 강했다" 고 한 항목 재점검

### 3.1 "Faithfulness 0.959 → paid `/answer` 불필요"

- 근거 약점: RAGAS Faithfulness 는 LLM-judge 기반. 사용자 실제 query 에서도 0.959 인지 미검증
- Codex 의 "답변 = 사용자 액션 only" 가 비용 통제 측면에서 더 일관
- **재판정**: Claude 의 권고("무료 Flash 자동 답변 유지")는 **절대적 근거 아님**. Codex 의 사용자 액션 only 가 페르소나 A 의 chunk-first UX (기억 보조) 와 더 정합

### 3.2 "Reranker free tier 한도 분석 (50→20 cap)"

- Claude senior-planner 가 HF Inference free tier ~1k call/day 한도 + candidate 50→20 cap 권고. Codex 는 "BGE reranker 무료" 만 표기
- **재판정**: Claude 우위 유지

### 3.3 "sweep × retry 정확 계산 (worst 9, 50p worst 450)"

- Claude senior-developer 가 코드 직접 분석. Codex 는 "재시도 중복" 정성적
- **재판정**: Claude 우위 유지

### 3.4 "P95 latency SLO 3s 정합성 분석"

- Claude 는 latency 우선순위, Codex 는 비용 우선순위 — 다른 axis
- **재판정**: 우열 X (관점 차이)

**원칙 (Codex 철학 채택)**:
1. 검색 기본 경로는 무료
2. 유료는 ① 무료 파서가 못 본 인제스트 보강 (vision) ② 검색 후보 만들기 강화 (query decomposition)
3. 모든 유료 호출은 cache key 필수 (page_hash + prompt_version)
4. 문서당·일별 비용 상한 (P0 추가)
5. 답변 생성은 사용자 액션 또는 복잡 질문 조건부

### 5.2 검색 파이프라인 (수정판)

```
사용자 질문
  ↓
NFC 정규화
  ↓
날짜/문서/포맷/태그 메타 필터 추출
  ↓
질문 유형 분류 (룰 기반 → fallback Flash-Lite)
  ↓ ┌── C5 (메타 필터: "지난달", "#태그") → 메타만, RAG skip → 0.5s 응답
    └── 일반 + 비교/인과/cross-doc → 다음 단계
  ↓
[NEW] Query decomposition (Flash-Lite default on, sub-query 2~5개)
  ↓
각 sub-query → dense + sparse + doc_embedding 병렬 검색
  ↓
RRF merge
  ↓
BGE reranker top 20→재정렬 (HF 무료, candidate cap)
  ↓
[NEW] MMR/diversity 중복 제거
  ↓
top-K chunk + page/section/table caption + 출처 표시
  ↓
(사용자 명시 액션 시만) Gemini Flash 답변 생성
```

**default on**: doc-level RRF, query expansion, **multi-query (Codex 채택)**, reranker, **MMR (Codex 채택)**
**조건부**: `/answer` (사용자 클릭 시), HyDE (off 유지)

## 7. Codex 에게 재검증받고 싶은 핵심 질문 9개

이 문서를 Codex 에 다시 의뢰할 때 답변 받고 싶은 검증 포인트.

### Q1. Multi-query default on 채택의 정합성

> Claude 가 1차로 escalation only 권고했다가 자기방어 인정 후 default on (Codex 권고) 으로 수정. 이 수정이 정합성 있나? 또는 trigger 조건 (intent 분류기로 cross-doc 신호 잡고 분기) 이 더 나은가?

특히 검토 부탁:
- Flash-Lite query 분해 비용 ~$0.30/월 추정의 정확성
- 페르소나 A "다양·복잡 질문" 가정이 Multi-query default on 을 정당화하나?
- intent 분류기 1회 호출 vs 일괄 분해 비용 trade-off

### Q2. `/answer` 자동→사용자 액션 only 전환의 페르소나 적합성

> Codex 권고대로 `/answer` 를 사용자 명시 액션 only 로 변경. 그러나 페르소나 A 가 "다양·복잡 질문 가능성 높음" 이라면 답변 자동 생성이 더 나은 UX 일 수도. trade-off 어떻게 보는가?

검토 부탁:
- 사용자가 chunk 만 보고 직접 종합 → 기억 보조 페르소나에 맞나?
- 답변 자동 생성을 막으면 "복잡 질문" 사용 시나리오에서 UX 저하?
- 절충안 (예: 짧은 자동 답변 + "자세히" 클릭 시 깊이 분석) 가치?

### Q3. Vision 운영 모드 기본값 5~10p 의 적정성

> 사용자 PDF 평균 페이지 수 미상 + 자료 분포 (현재 11 docs: PDF 5건 중 페이지수 알려진 것 = sonata 99c, 데이터센터 41p, sample-report 1045p…). 기본 5~10p 가 적정한가?

검토 부탁:
- vision_need_score 임계값 (예: text_density < 0.3 + image_area_ratio > 0.4) 의 구체 기준?
- 운영 모드 분기를 사용자가 매 인제스트마다 선택 vs doc_type 별 default?

### Q4. 인제스트 보조 인덱스 7항목 우선순위

> Codex 권고 7항목 (요약/section 요약/표 caption/그림 caption/엔티티/동의어/예상 질문) 모두 ship 가치 있나? 어디서 자르는 게 ROI 정점?

검토 부탁:
- 표/그림 caption 은 vision 호출과 같이 추출 → 거의 무료. P1
- 엔티티/동의어 은 별도 LLM 호출 필요 → 비용 발생
- 예상 질문은 가치 검증 필요 (실제 사용자 query 와 매칭률)

### Q5. Multi-query 분해 모델 — Flash-Lite vs Flash

> Codex 권고가 "복잡 질문일 때 Flash-Lite 또는 Flash" 로 모호. 한국어 cross-doc 의도 정확 분해에 Flash-Lite 가 충분한가?

검토 부탁:
- Flash-Lite 한국어 분해 정확도 검증 사례
- 분해 quality 가 검색 정확도에 미치는 영향 (잘못 분해되면 검색 노이즈 ↑)
- 비용 차이 vs quality 차이 trade-off

### Q6. HWP/HWPX vision 미적용 — Codex 가 빠뜨린 영역

> 현재 적재 11 docs 중 hwp/hwpx 4건 (한컴 5.x, 운영내규, 직제규정 등). vision_enrich 는 PDF only. 한컴 포맷의 그림/표는 검색에서 누락 중. Codex 가 이 영역에 대한 권고 안 함.

검토 부탁:
- 한컴 포맷 vision 적용 가치 (사용자 자료 한국어 도메인 비중 高)
- 구현 path (PyHWP 이미지 추출 → Vision 호출) 비용 추정

### Q7. 문서당 비용 상한 메커니즘 (F6) 구체 설계

> Codex 가 "문서당·사용자당·일별 비용 상한" 권고했지만 구체값·구현 방식 미제시.

검토 부탁:
- 문서당 cap (예: $0.05/doc) 적정값?
- 일별 cap 도달 시 동작 (인제스트 거부 vs 무료 fallback)?
- 사용자 가시성 (현재 누적 비용 표시 UI 필요)?

### Q8. Sprint 순서 정합성 (S0 → S1 → S2 → S3 → S4)

> 위 §6 의 sprint 순서가 적절한가? 다른 순서 (예: S3 인제스트 보조 인덱스가 S2 검색 path 보다 먼저) 가 ROI 더 높을 가능성?

검토 부탁:
- 인제스트 보조 인덱스 (표/그림 caption) 가 검색 reranker 보다 정확도 영향 큰가?
- S0 (비용 누수 fix) 후 S1 (골든셋 + 검색 stack) vs S3 (인제스트 보조) 우선순위?

### Q9. 사용자 실 query 데이터 누적 plan

> Claude 1차 권고에 (E) "실 query 대시보드" 가 있었는데 Codex 답변에 명시 없음. `search_metrics_log` + `answer_feedback` 이미 있는데 시각화·활용 plan 가치는?

검토 부탁:
- 자동 골든셋 N=100+ 확장 vs 실 query 데이터 누적 — 어느 쪽 통계 신뢰성 ↑?
- 실 query 분포 → 자동 골든셋 카테고리 분포 보정 메커니즘?

## 8. 메타 — Claude 의 자기방어 frame 학습

### 8.1 자기방어 패턴 식별

Claude 가 1차 응답에서 사용한 frame:
- "일치 9 / Codex 우위 6 / **Claude 우위 4** / 충돌 1"
- 충돌 1건을 "**우리가 더 신중하다**" 로 결론
- "**Self-defensible**" 단어 사용으로 답변 강화

→ **자기 진영 옹호 bias**. 비교 분석에서 "내 답이 더 나음" 으로 frame 짜는 경향.

### 8.2 사용자 push 가 만든 교정

사용자가 "100% 확신해? 비판적으로 봐" 한 push 가 없었으면 자기방어 frame 그대로 sprint 진입.

### 8.3 향후 패턴

- Claude vs 외부 의견 비교 시 자체 비판 N회 강제 (Karpathy 가이드라인 적용)
- "내 답이 더 신중·합리적" 결론 시 근거 4개 모두 측정 가능 데이터인지 점검
- 모델 가정 (Flash vs Flash-Lite) 같은 frame 차이 우선 확인
