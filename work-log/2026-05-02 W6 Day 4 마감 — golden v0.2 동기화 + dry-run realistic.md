### 4.1 W6 Day 2 발견 사항

### 4.2 회수 — `realistic` 모드

```python
def _realistic_rechunk(doc: dict) -> tuple[list, str | None]:
    """Storage 에서 원본 다운로드 + 파서 실행 → 실 인제스트와 동일한 입력 분포로
    chunk policy 통과."""
    parser = _PARSERS_BY_DOC_TYPE.get(doc["doc_type"])
    storage = SupabaseBlobStorage(bucket=...)
    data = storage.get(doc["storage_path"])
    if doc["doc_type"] == "hwp" and is_hwpml_bytes(data[:4096]):
        parser = _hwpml_parser
    extraction = parser.parse(data, file_name=...)
    sections = list(extraction.sections)
    split = _split_long_sections(sections)
    merged = _merge_short_sections(split)
    return _to_chunk_records(doc_id="dryrun-doc", sections=merged), None
```

## 5. 비판적 자가 검토

1. **G-020 정정의 의도 변화**: 원 의도는 "Day 4 인제스트한 작은 샘플 회귀 안전망". 4d0ea2c4 가 손상 데이터로 안전망 역할 부적절 → expected 변경. golden v0.3 의 정식 버전 갱신은 별도 작업 (현 변경은 batch script 의 in-line micro-revision).
2. **realistic 모드 비용**: 6 doc 처리에 ~30초 (Storage I/O + 파서). 자료 누적 시 (50+ doc) ~5분 가능. CI 도입 시 simulated 가 적합.
3. **realistic 모드의 idempotency 검증**: Δ=0% 결과는 chunk policy 가 stable (재인제스트 시 동일 결과) 임을 확인. chunk policy 추가 변경 시 동일 검증 절차 권장.
4. **md doc 미지원**: jet_rag_sample / jrag_day4 (md) 는 `_PARSERS_BY_DOC_TYPE` 에 없음 → `unsupported doc_type=md` 에러. md 파서 도입 시 자동 활성. 현재는 chunks=0 으로 영향 0.
5. **DOCX 포함 검증**: realistic 모드가 DocxParser 자동 디스패치 — 향후 사용자 DOCX 업로드 시 동일 절차로 dry-run 가능.
