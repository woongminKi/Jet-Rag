### 1.1 아키텍처 B 정밀 명세 (httpOnly 함정 회피 — 2026-05-20)

프론트(`jetrag.woong-s.com`)와 API(`jetrag-api.woong-s.com`)가 **분리된 형제 서브도메인**이고, 프론트가 서버 컴포넌트(`page.tsx await getStats()`)와 클라이언트 컴포넌트 양쪽에서 API를 직접 호출한다. `@supabase/ssr` 의 **표준 browser client 패턴은 쿠키를 JS-readable 로 두므로 httpOnly 가 아니다**(localStorage 동급). 진짜 httpOnly 를 얻으려면 다음 하이브리드가 필수:

- **세션은 서버 관리(httpOnly)**: 로그인/가입/로그아웃/OAuth 콜백을 Next.js **server action 또는 route handler** 로 처리 → `@supabase/ssr` `createServerClient` 가 httpOnly 쿠키 set. browser client 로 세션을 관리하지 않음.
- **세션 리프레시**: `middleware.ts` 가 서버에서 갱신 + httpOnly 쿠키 재set.
- **쿠키 도메인**: prod `domain=.woong-s.com` (형제 서브도메인 공유), dev 는 domain 생략(host-only localhost). secure(prod)·sameSite=lax.
- **데이터 콜(클라이언트)**: `fetch(url, { credentials: 'include' })` → 브라우저가 httpOnly 쿠키 자동 첨부 → **JS 가 토큰을 만지지 않음(진짜 httpOnly)**. 추가 네트워크 홉 없음(P95 보호).
- **데이터 콜(서버 컴포넌트)**: `next/headers cookies()` 로 세션 읽어 `Authorization: Bearer` 또는 Cookie 헤더로 jetrag-api 에 forward.
- **백엔드**: `get_current_user` 가 **Authorization 헤더 + Supabase auth 쿠키(`sb-<ref>-auth-token`) 양쪽**에서 토큰 추출. CORS `allow_credentials=True` + **명시 origin**(와일드카드 금지) — `JETRAG_CORS_ORIGINS` 재사용.
- **UI auth 상태**: 서버 컴포넌트가 초기 user 를 client 로 내려주고, 클라이언트는 `/auth/me`(또는 server action) 로 상태 동기화. browser client `getSession()` 의존 안 함(httpOnly 라 못 읽음).

> 작업량 재추정: Phase B ~3~4h (서버 관리 auth 가 표준 client 패턴보다 무거움). P95 영향 0(데이터 콜 직접 + credentials).

## 5. 초대 코드 마이그 017 + 가입 게이트

```sql
-- 017_invite_codes.sql
CREATE TABLE IF NOT EXISTS invite_codes (
    code        TEXT PRIMARY KEY,
    issued_by   UUID,
    used_by     UUID,
    used_at     TIMESTAMPTZ,
    expires_at  TIMESTAMPTZ,
    note        TEXT,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX IF NOT EXISTS idx_invite_codes_unused
    ON invite_codes (code) WHERE used_by IS NULL;
ALTER TABLE invite_codes ENABLE ROW LEVEL SECURITY;
```

- 검증: `code` 존재 AND `used_by IS NULL` AND (`expires_at IS NULL OR expires_at > now()`)
- 소진(race 방어): `UPDATE invite_codes SET used_by=$user, used_at=now() WHERE code=$code AND used_by IS NULL` → affected=0 이면 이미 소진
- 가입 직후 검증(D1-Q5): signUp 성공 → POST `/auth/redeem-invite {code}` → 실패 시 프론트 즉시 signOut + 안내
- Google OAuth: "invite 소진 이력 없는 user = 미승인" → 콜백 후 redeem 통과 못 하면 차단
- 베타 30개 seed SQL 스니펫 포함
