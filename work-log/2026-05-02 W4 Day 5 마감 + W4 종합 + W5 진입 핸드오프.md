## 6. W4 운영 정책 (변경/추가)

### 6.1 graceful degrade 패턴 — W4 표준

W3 Day 5 의 HwpxDocument 패턴이 W4-Q-17 (PyMuPDFParser dict → blocks fallback) 에 재사용. W5 의 DOCX/PPTX 파서에도 동일 적용 권장.

### 6.2 인메모리 LRU 캐시 패턴

W4 Day 1 의 `BGEM3HFEmbeddingProvider` LRU (maxsize + Lock + defensive copy) 가 W6+ Redis 도입 전 기본 패턴. 다른 외부 호출 (Gemini Flash 등) 도 동일 적용 가능 (W6+ 검토).

### 6.3 dry-run before 본 적용 — W4 표준

W4-Q-14·15 모두 dry-run 리포트 작성 후 본 적용. DE-65 처럼 본 적용 보류 결정도 가능. R-W3-4 정성 검토 게이트 패턴 재활용.

### 6.4 worst-case + 정상 시나리오 비교 dry-run

W4-Q-14 dry-run 의 정상 시나리오 (Δ=0%) + worst-case (-66.8%) 동시 보고 패턴. W5 의 청킹 정책 추가 변경 시 동일 적용.

### 6.5 senior-developer + 메인 스레드 trust-but-verify

W4 Day 3 의 chunk.py 변경에서 senior-developer 산출물의 leftmost 매칭 버그를 메인 스레드 직접 smoke 테스트로 발견. CLAUDE.md trust-but-verify 정합 — 에이전트 위임 후 직접 검증 필수.

### 6.6 비판적 재고 — 모든 명세에 적용

W4 명세의 비판적 재고가 4.3 W5 이월 + DE-65 본 적용 W6+ 등 안전한 결정으로 이어짐. W5 진입 시도 동일 패턴 유지.
