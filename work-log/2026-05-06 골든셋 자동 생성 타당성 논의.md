# 2026-05-06 골든셋 자동 생성 타당성 논의

## 배경

Jet-Rag 의 검색 품질 평가(Recall@K, Ragas Faithfulness, 출처 일치율)를 위해서는 (질문, 정답, 출처) 3-tuple 로 구성된 골든셋(평가셋)이 필요하다. 사용자는 "업로드한 파일에 대해 LLM(Claude)이 골든셋 템플릿을 자동 생성"하는 워크플로를 검토 중이다.

이 문서는 워크플로 시작 전, **"LLM 이 단일 문서로부터 production-grade 골든셋을 만들 수 있는가"** 에 대한 비판적 타당성 검토 대화를 정리한다.

---

## 1. 사용자가 원한 워크플로 (초기 요청)

> Jet-Rag 는 유저가 업로드한 파일에 대해서 시간이 지나 나중에 어떠한 질문을 했을 때 해당 질문에 대해서 가장 적합한 데이터를 보여주는 '기억 보조 장치' 프로그램이다.
>
> 파일을 업로드하면 그 파일에 대한 골든셋 템플릿을 만든다.
> - 업로드한 파일을 분석한다.
> - 몇 가지 질문에 대해 품질 좋은 답변을 작성한다.
> - 복잡한 질문이 올 수 있다는 가정 하에 대답을 작성한다.
> - 질문이 해당 문서에 없으면 억지로 답변하지 말고, 해당 문서에 없다고 답한다.

## 2. 사용자의 후속 질문

> 내가 너한테 문서를 주면 너가 골든셋을 만들어 줄 수 있어? 무조건 가능하다고 하지 말고 비판적으로 생각하고 답해.

→ 사용자는 "무조건 가능"이라는 답을 차단하고, **현실적 한계까지 포함한 평가**를 요구.

---

## 3. 결론 — 한 줄 요약

> **"단일 문서 기반 골든셋 초안"은 만들 수 있다. 단, "production-grade 골든셋"은 LLM 단독으로 만들 수 없다.**

LLM 출력은 "**드래프트**"로 취급해야 하며, 사용자(또는 다른 검증 단계)의 후처리가 반드시 필요하다.

---

## 4. LLM 이 잘할 수 있는 것

| 항목 | 설명 |
|---|---|
| 사실 추출 | 문서에서 명시적 사실(fact) 을 찾아 (질문, 정답, 인용구) 3-tuple 로 변환 |
| 라벨링 | 난이도/유형(단답·다단계·요약·범위 외) 분류 |
| Negative 샘플 | "문서에 없음" 케이스를 의도적으로 끼워 넣어 out-of-scope 평가 가능 |
| 인용 강제 | 정답에 출처 인용구를 첨부 → 추후 출처 일치율 평가에 활용 |

## 5. 구조적 한계 (5가지)

### 5.1 질문 현실성 갭 ★ (가장 큰 위험)

LLM 이 만든 질문은 **"시험 문제" 톤**이 된다. 키워드가 너무 명확하고 문법이 정돈돼 있어 retrieval 이 쉽게 풀린다.

반면 실제 Jet-Rag 사용자(페르소나 A)는 며칠~몇 달 뒤 **반쯤 기억나는 자연어 질문**을 던진다.

> 예) "그때 그 회의에서 X 정했던 거 뭐였지...", "지난번에 정리한 그 파일에서 비용 얼마였더라"

이런 질문은 **시제·생략·동의어·오기억** 이 섞여 있어 LLM 이 시뮬레이션하기 어렵다.

**영향**: 이렇게 만든 골든셋으로 평가하면 **Recall@K 가 과대평가**된다. → 실제 운영 환경에서 검색이 깨져도 평가셋에서는 안 잡힘.

### 5.2 난이도 캘리브레이션 한계

LLM 은 "복잡한 질문" 을 multi-hop 으로 만들 수 있지만, RAG 가 **실제로 실패하는** 케이스는 다음과 같다:
- 어휘 불일치 (질문어 ≠ 문서어)
- 동의어/유의어 치환
- 표 안의 숫자 (chunk 분할로 컨텍스트 손실)
- 시간 경과로 인한 표현 변형

이 케이스들은 **시스템을 돌려보지 않으면** 어디서 깨지는지 모른다. LLM 머릿속의 "어려움" ≠ 검색기의 "어려움".

### 5.3 자가 채점 편향

LLM 이 정답까지 작성하면, 그 정답으로 평가할 때 **순환 검증** 이 발생한다.

특히 Ragas Faithfulness 같은 LLM-as-judge 평가에서 같은 모델 계열을 쓰면 점수가 부풀려질 위험.

**완화책**: 정답 작성자 ≠ 평가자(다른 모델 또는 사람), 또는 정답을 "직접 인용구"로만 제한.

### 5.4 모달리티 손실

파일이 다음 중 하나면 LLM 의 파싱 충실도가 떨어진다:
- 스캔 PDF (OCR 의존)
- 표/차트 위주
- 다이어그램·도식
- 이미지 안의 텍스트

→ 이런 파일에서는 그림에만 있는 정보가 골든셋에서 **누락**된다. 이는 Jet-Rag 에서 특히 위험한데, 실제 사용자가 "그 다이어그램에서 봤던 X" 같은 질문을 던지기 때문.

### 5.5 단일 문서 한계

실제 운영 골든셋은 **코퍼스 전반의 cross-doc 질문**을 포함해야 한다.

> 예) "파일 A 의 회의록에서 정한 정책이 파일 B 의 가이드라인에 어떻게 반영됐지?"

한 파일만 주면 이런 케이스를 만들 수 없다. → 운영 환경에서 가장 어려운 검색 시나리오를 평가에서 빠뜨린다.

## 6. 권장 운영 방식

| 단계 | 담당 | 산출물 |
|---|---|---|
| 1. 드래프트 생성 | LLM (Claude) | (질문, 정답, 인용) 3-tuple 초안 N건 |
| 2. 질문 자연어화 | 사용자 | 시험 문제 톤 → 페르소나 A 톤으로 rephrasing |
| 3. 어려운 케이스 추가 | 사용자 | 실제로 헷갈렸던 상황을 시드로 제공 → LLM 이 변형 |
| 4. cross-doc 보강 | 사용자 + LLM | 여러 파일을 같이 주고 cross-doc 질문 생성 |
| 5. 모달리티 검증 | 사용자 | 표/그림에 있는 사실이 누락 안됐는지 수동 확인 |

### 핵심 원칙

- LLM 출력 = **드래프트**, 절대 production 으로 직행 X
- "복잡한 질문" 카테고리는 **자동 생성보다 사용자 시드 기반 변형**이 더 현실적
- 가능하면 **여러 파일을 같이 입력**해 cross-doc 케이스 확보
