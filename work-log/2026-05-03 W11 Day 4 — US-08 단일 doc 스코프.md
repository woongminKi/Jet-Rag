## 1. 비판적 재검토

### 1.1 옵션

| 옵션 | 설계 | 결정 |
|---|---|---|
| A | search_hybrid_rrf RPC 에 doc_id 인자 추가 | ❌ 마이그레이션 필요 + DB 기반 변경 |
| **B** | 응용 layer 필터 (RPC 결과 후 doc_id 일치만 보존) | ✅ 채택 — 마이그레이션 회피 |
| C | 신규 `/documents/{id}/qa` 엔드포인트 | ⚠ RESTful design ↑ but 코드 중복 |

→ **B 채택**. trade-off: RPC 가 N건 결과 반환 후 일부만 통과 → top-K cap 안에서 부족할 수 있으나, 실제 doc_id 별 chunks N개 분포 상 충분.

### 1.2 형식 검증

```python
if doc_id is not None:
    doc_id = doc_id.strip()
    if not doc_id or len(doc_id) > 64:
        raise HTTPException(400, "doc_id 형식이 유효하지 않습니다.")
```

응용 layer 필터링이라 SQL injection 위험 0이지만 보수적 검증 — 빈 문자열 / 너무 긴 입력 차단.

if doc_id is not None:
    doc_id = doc_id.strip()
    if not doc_id or len(doc_id) > 64:
        raise HTTPException(400, ...)

# 3. RPC 후 응용 layer 필터
if doc_id is not None:
    rpc_rows = [r for r in rpc_rows if r.get("doc_id") == doc_id]

# dense_hits / sparse_hits / fused 카운트 자동 갱신
