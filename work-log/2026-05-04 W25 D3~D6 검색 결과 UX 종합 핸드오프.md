### 4.2 변경 (`api/app/routers/search.py`)

```python
# 상수
_COVER_GUARD_TEXT_LEN = 30
_COVER_GUARD_PENALTY = 0.3

# 누적 단계
def _is_cover_chunk(meta) -> bool:
    return (
        meta["text_len"] <= _COVER_GUARD_TEXT_LEN
        and (meta["chunk_idx"] == 0 or meta["page"] == 1)
    )

# RPC 결과 후처리: 가드 매칭 시 score *= 0.3
```

추가 fetch: candidate chunks metadata SELECT IN-list 1회 (PK 인덱스 lookup, ~5~15ms).

### 4.3 신규

- `api/tests/test_search_cover_guard.py` — 4건 (3 핵심 + 1 상수)
- 회귀 보호 — 짧은 본문 헤딩 (예: chunk_idx=10 + page=5 + text_len=15 "결론") false positive 0 검증

### 4.4 효과 측정 (실데이터)

```sql
-- 가드 매칭 분포 (SONATA 99 chunks)
SELECT count(*) AS total, count(*) FILTER (WHERE matches) AS guard_match
FROM (SELECT length(text) <= 30 AND (chunk_idx = 0 OR page = 1) AS matches FROM chunks WHERE doc_id = ?);
```

| 지표 | 값 |
|---|---|
| 가드 매칭 | **1/99 (1.0%)** — 매우 보수적 |
| DOCX false positive (`page IS NULL + chunk_idx=0`) | 0건 |
| 단일 PDF p.1 본문 false positive | 0건 |
| 유일한 매칭 | `chunk_idx=0 / page=1 / text_len=6 / "SONATA"` ← **정확히 의도한 표지** |

## 7. W25 D6 — 매칭 청크 % 표시 통일

### 7.1 사용자 결정

> "관련도 순이 어떤걸 기준으로 한건지 잘 모르겠어. 12번 이미지처럼 '%' 형식으로 보여줄 수 있나?"

### 7.2 변경 (`web/src/app/doc/[id]/page.tsx` MatchedChunksSection)

| Fix | 변경 |
|---|---|
| 1. 라벨 통일 | `관련도순` → `매칭 강도순 ⓘ` (검색 결과 카드와 일관) |
| 2. % 표시 | `RRF 0.0161` → `매칭 강도 N%` (top chunk 대비 정규화) |
| 3. raw RRF 제거 | 사용자 친화 우선, debug 토글은 별도 메커니즘 |

정규화 공식:
```ts
matchPct = Math.max(1, round(chunk.rrf_score / chunks[0].rrf_score * 100))
```
0% 회피 위해 최소 1% 보장. 백엔드가 score desc 정렬해서 chunks[0] = max 보장.

섹션 헤더 ⓘ 툴팁: "이 문서 안 청크들 중 가장 강한 매칭 대비 상대 강도예요. 정답 신뢰도와는 다릅니다."

### 7.3 멘탈 모델 일관성

| 위치 | 라벨 | 단위 | 100% 의미 |
|---|---|---|---|
| 검색 결과 카드 | `매칭 강도 100%` + 막대 | doc 단위 | 결과 집합 내 최강 doc |
| doc 페이지 청크 | `매칭 강도 100%` 텍스트만 | chunk 단위 | 이 doc 안 최강 청크 |

라벨 동일 → 학습 비용 0. 단위 차이는 헤더 ⓘ 툴팁으로 안내.
