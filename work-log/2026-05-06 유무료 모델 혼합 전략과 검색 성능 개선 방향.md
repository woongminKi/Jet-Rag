# 2026-05-06 유무료 모델 혼합 전략과 검색 성능 개선 방향

> 프로젝트: Jet-Rag
> 작성일: 2026-05-06
> 목적: 유료 LLM 비용을 통제하면서 검색 성능을 높이기 위한 모델 배치, 파이프라인, 골든셋 운영 방향 정리

---

## 0. 한 줄 결론

> **유료 LLM 을 검색 전체에 깔지 말고, 무료 retrieval 이 실패하기 쉬운 지점에만 제한적으로 써야 한다.**

Jet-Rag 의 핵심 가치는 "유저가 문서를 업로드해두고, 시간이 지난 뒤 복잡하거나 흐릿한 질문을 했을 때 빠르게 근거 chunk 를 찾아 간략히 보여주는 것"이다.

따라서 기본 검색 경로는 무료/저비용 모델과 로컬 파서 중심으로 유지하고, 유료 모델은 다음 두 지점에 집중하는 것이 적절하다.

1. **인제스트 보강**: 무료 파서가 못 보는 표·그림·스캔·다이어그램
2. **복잡 질문 처리**: 질문 분해, 라우팅, 제한적 답변 생성

## 1. 현재 비용 상황에 대한 판단

### 1.1 사용자 관찰

- 지금까지 RAG 관련 주요 모델은 무료 모델 위주로 사용했다.
- 이미지/표 파싱에만 Gemini 유료 모델을 적용했다.
- 100페이지 PDF 중 50페이지 정도만 vision 처리하도록 제한했다.
- 약 10회 테스트 후 2만원 충전액 중 약 8천원이 사용되었다.
- 체감상 비용이 예상보다 많이 발생하고 있다.

### 1.2 의심 지점

현재 코드 구조상 비용이 커질 수 있는 원인은 모델 단가 자체보다 다음 쪽을 먼저 의심해야 한다.

| 원인 | 설명 |
|---|---|
| 재시도 중복 | Gemini 공통 retry 3회 + vision enrich sweep 최대 3회 구조로 실패 페이지가 여러 번 호출될 수 있음 |
| 같은 문서 반복 reingest | page hash 기반 vision cache 가 없으면 동일 페이지를 매번 다시 과금 |
| 출력 과다 | OCR 전체 텍스트 + caption + structured JSON 을 모두 생성하므로 output token 이 커질 수 있음 |
| 큰 이미지 입력 | PDF 페이지를 150 DPI 로 렌더링 후 이미지 입력. 이미지 토큰이 페이지 크기/타일 수에 따라 커질 수 있음 |
| usage 기록 부재 | Gemini `usage_metadata` 를 저장하지 않아 실제 input/output/thinking token 을 추적하기 어려움 |
| "첫 50페이지" 정책 | 정보량과 무관하게 앞쪽 페이지를 일괄 처리해 비용 대비 효율이 낮을 수 있음 |

### 1.3 즉시 필요한 비용 가시성

Gemini 호출마다 다음 값을 기록해야 한다.

- model
- source_type: `pdf_vision_enrich`, `pdf_scan`, `image`, `pptx_rerouting` 등
- doc_id
- page
- prompt_tokens
- image_tokens 또는 input_tokens
- output_tokens
- thinking_tokens 가 노출되는 경우 해당 값
- retry_attempt
- success/failure
- estimated_cost

이 값이 없으면 "어디서 돈이 새는지"를 찾기 어렵다.

공식 가격/토큰 기준 확인 문서:

- [Gemini API pricing](https://ai.google.dev/gemini-api/docs/pricing?hl=en)
- [Gemini token counting](https://ai.google.dev/gemini-api/docs/tokens?hl=en)

## 2. 모델 배치 원칙

### 2.1 기본 원칙

1. **검색의 기본 경로는 무료로 유지한다.**
2. **유료 모델은 recall 을 직접 올리는 곳보다, 검색 후보를 더 잘 만들거나 무료 파서가 못 본 정보를 보강하는 곳에 쓴다.**
3. **유료 호출은 항상 cache key 를 가져야 한다.**
4. **문서당/사용자당/일별 비용 상한을 둔다.**
5. **답변 생성은 기본 자동이 아니라 사용자 액션 또는 복잡 질문 조건부로 둔다.**

### 2.2 권장 모델 배치표

| 작업 | 기본 모델/방식 | 유료 사용 조건 |
|---|---|---|
| 텍스트 PDF 파싱 | PyMuPDF | 사용 안 함 |
| HWP/HWPX 파싱 | 로컬 파서 | 사용 안 함 |
| DOCX/PPTX 텍스트 파싱 | python-docx / python-pptx | 텍스트 없는 슬라이드만 vision |
| 이미지/스캔 OCR | Gemini Vision 제한 호출 | 후보 페이지/이미지만 |
| 표/그림/다이어그램 보강 | 페이지 선별 후 Gemini Vision | 중요 페이지 또는 사용자 고품질 모드 |
| chunk embedding | BGE-M3 무료/HF 또는 로컬 | 가능하면 사용 안 함 |
| query embedding | BGE-M3 + LRU cache | 사용 안 함 |
| sparse 검색 | PGroonga/Mecab | 사용 안 함 |
| query expansion | 룰/사전 기반 | 복잡 질문일 때 Flash-Lite |
| query decomposition | 기본 off | 복잡 질문일 때 Flash-Lite 또는 Flash |
| reranking | BGE reranker 무료 | 가능하면 사용 안 함 |
| 답변 생성 | 검색 결과/청크 표시 | 사용자가 답변 생성 요청 시 Flash-Lite/Flash |
| 골든셋 draft 생성 | LLM draft | production 직행 금지, 사용자 검수 필수 |

## 3. PDF Vision 정책 변경 제안

### 3.1 현재 방식의 문제

현재 `JETRAG_PDF_VISION_ENRICH_MAX_PAGES=50` 방식은 "앞에서부터 50페이지"를 처리한다.

이 방식은 단순하지만 다음 문제가 있다.

- 표/그림이 뒤쪽에 있으면 놓친다.
- 앞쪽에 텍스트 페이지가 많으면 비용을 낭비한다.
- 100페이지 PDF 를 여러 번 테스트하면 비용이 선형으로 증가한다.
- 실패/재시도까지 겹치면 실제 호출 수가 예상보다 커질 수 있다.

### 3.2 권장 방식: 후보 페이지 선별

무조건 앞 50페이지가 아니라, PyMuPDF 로 모든 페이지를 먼저 무료 분석한 뒤 vision 후보를 고른다.

```text
PDF 업로드
→ PyMuPDF 로 전체 페이지 무료 분석
→ 페이지별 vision_need_score 계산
→ 상위 N 페이지 선별
→ Flash-Lite Vision 우선 호출
→ 결과 빈약/복잡 페이지에만 Flash 재시도
→ page_hash + prompt_version 으로 결과 캐시
→ 이후 reingest 에서는 cache hit 시 Gemini 호출 0
```

### 3.3 후보 페이지 점수 기준

| 신호 | 의미 |
|---|---|
| text_density 낮음 | 스캔/이미지 위주 페이지 가능성 |
| image_area_ratio 높음 | 그림/다이어그램/스캔 가능성 |
| table_like_blocks 존재 | 표 정보 보강 필요 가능성 |
| PyMuPDF text extraction 품질 낮음 | 텍스트 순서 깨짐 또는 누락 가능성 |
| figure/table caption 존재 | 주변 페이지 vision 가치 높음 |
| 페이지 내 숫자/단위/기관명 밀도 높음 | 질문 대상이 될 가능성 높음 |

### 3.4 운영 모드

| 모드 | vision 페이지 상한 | 용도 |
|---|---:|---|
| 기본 | 5~10 pages/doc | 비용 통제, 일반 사용자 기본값 |
| 품질 우선 | 20~30 pages/doc | 중요한 문서, 사용자가 명시 선택 |
| 전체 고품질 | manual only | 데모/평가/특수 문서 |

기본값은 `최대 10페이지`가 적절하다.

## 4. 검색 성능 향상 파이프라인

### 4.1 방향

복잡한 질문을 잘 처리하려면 유료 LLM 으로 전체 답변을 바로 생성하는 것보다, 검색 전후 구조를 강화하는 편이 비용 대비 효과가 좋다.

핵심은 다음이다.

- query 를 더 잘 이해한다.
- sub-query 로 나눈다.
- dense/sparse/doc-level 후보를 넓게 모은다.
- reranker 로 순서를 바로잡는다.
- 중복 chunk 를 제거하고 다양한 근거를 보여준다.

### 4.2 권장 검색 흐름

```text
사용자 질문
→ NFC 정규화
→ 날짜/문서/포맷/태그 필터 추출
→ 질문 유형 분류
→ 복잡 질문이면 sub-query 2~5개로 분해
→ 각 sub-query 를 dense + sparse + doc_embedding 으로 검색
→ RRF merge
→ BGE reranker 로 top 30 재정렬
→ MMR/diversity 로 중복 제거
→ 상위 chunk + page/section/table caption 표시
→ 필요할 때만 LLM 답변 생성
```

### 4.3 유료 모델을 쓸 만한 검색 지점

검색 성능 향상에서 유료 모델을 쓸 만한 1순위는 **query decomposition** 이다.

예를 들어 사용자가 다음처럼 묻는 경우:

> "A 문서에서 말한 정책이 B 자료의 예산 조건이랑 어떻게 연결돼?"

단일 검색어로는 어렵다. 이때 유료 LLM 이 답변을 바로 만들기보다, 다음처럼 검색 가능한 질문으로 쪼개는 역할을 맡는 것이 좋다.

```json
{
  "intent": "cross_doc_compare",
  "sub_queries": [
    "A 문서 정책 핵심 내용",
    "B 자료 예산 조건",
    "정책과 예산 조건의 연결점"
  ],
  "filters": {
    "doc_scope": ["A", "B"]
  }
}
```

이후 실제 검색은 무료 retrieval 엔진이 수행한다.

## 5. 인제스트 시점에 저장하면 좋은 검색 보조 인덱스

Jet-Rag 는 "나중에 빠르게 찾아 보여주는 앱"이므로, 업로드 시점에 검색 보조 정보를 만들어두면 좋다.

| 저장 항목 | 목적 |
|---|---|
| 문서 3줄 요약 | doc-level search, 결과 카드 요약 |
| section 요약 | 긴 문서의 특정 주제 검색 |
| 표 caption | 표 검색 가능성 향상 |
| 그림/다이어그램 caption | vision 정보 검색 가능성 향상 |
| 핵심 엔티티 | 기관명, 제품명, 날짜, 금액, 수치 검색 |
| 동의어 후보 | 쏘나타/Sonata, 휴관/휴무 등 |
| 예상 질문 | 나중에 물을 법한 질문을 검색 보조 텍스트로 활용 |

단, 이 작업을 모두 비싼 모델로 하면 안 된다.

권장 배치는 다음과 같다.

- 문서/section 요약: Flash-Lite 또는 무료 모델
- 엔티티 추출: 룰 + 작은 모델
- 동의어 후보: 도메인 사전 + 필요한 경우 작은 LLM
- 표/그림 caption: 선별 페이지에만 Vision
- 예상 질문: 평가/검색 보조용으로 제한 생성

## 6. 골든셋 운영 방향

### 6.1 결론

골든셋은 반드시 필요하다.

하지만 LLM 자동 생성만으로 production-grade 골든셋을 만들면 안 된다. LLM 이 만든 질문은 "시험 문제"처럼 명확해져 실제 사용자의 흐릿한 기억 기반 질문을 충분히 반영하지 못한다.

따라서 LLM 골든셋은 **draft** 로 보고, 사용자 검수와 자연어화가 들어가야 한다.

### 6.2 권장 스키마

```csv
id,query,query_type,expected_answer,expected_doc_ids,relevant_chunks,acceptable_chunks,negative,notes
G-001,"그때 쏘나타 시트 뭐였지?",fuzzy_memory,"인조가죽/천연가죽/Nappa...",doc1,"22,23","21,24",false,"사용자 기억형"
G-002,"이 문서에 테스트베드 사업 체계 그림 있어?",vision_diagram,"p.6 다이어그램에 있음",doc2,"45","46",false,"그림 기반"
G-003,"운영내규랑 직제규정에서 위원회 역할 차이?",cross_doc,"A는..., B는...",doc3|doc4,"12|7","13|8",false,"cross-doc"
G-004,"여기에 개인정보 보관기간 나와?",out_of_scope,"문서에 없음",doc5,"","",true,"부정 질문"
```

### 6.3 query_type 권장 분류

| query_type | 설명 |
|---|---|
| `exact_fact` | 문서에 명시된 사실 질의 |
| `fuzzy_memory` | 사용자가 일부만 기억하는 질문 |
| `synonym_mismatch` | 질문어와 문서어가 다른 케이스 |
| `numeric_lookup` | 숫자/금액/비율/날짜 검색 |
| `table_lookup` | 표 안의 값 검색 |
| `vision_diagram` | 그림/다이어그램 기반 질문 |
| `summary` | 특정 문서/섹션 요약 |
| `cross_doc` | 여러 문서 비교/연결 |
| `out_of_scope` | 문서에 없는 질문 |

### 6.4 중요한 원칙

- `relevant_chunks` 하나만 두지 말고 `acceptable_chunks` 를 둔다.
- 실제 RAG 에서는 같은 사실이 여러 chunk 에 흩어져 있을 수 있다.
- LLM draft 질문은 사용자가 페르소나 A 톤으로 다시 바꾼다.
- cross-doc 질문과 negative 질문을 반드시 포함한다.
- 표/그림 관련 질문은 수동 검수가 필요하다.

## 7. 우선순위 제안

### P0 — 비용 누수 차단

1. Gemini `usage_metadata` 저장
2. Vision 호출별 estimated cost 계산
3. page hash + prompt version 기반 vision cache
4. 같은 문서 reingest 시 vision 재호출 금지
5. retry/sweep 별 실제 호출 수 로깅

### P1 — Vision 비용 구조 개선

1. "앞 50페이지" 정책 제거
2. PyMuPDF 기반 vision 후보 페이지 선별
3. 기본 vision cap 을 5~10 pages/doc 로 축소
4. Flash-Lite 우선, 실패/복잡 페이지에만 Flash 사용

### P2 — 검색 성능 개선

1. query decomposition 도입
2. doc_embedding RRF 기본 후보화 검토
3. BGE reranker cache 강화
4. synonym expansion 사전 확장
5. MMR/diversity 로 중복 chunk 제거

### P3 — 골든셋 운영

1. LLM draft 골든셋 생성
2. 사용자 자연어화
3. `query_type` 분포 맞추기
4. `relevant_chunks` + `acceptable_chunks` 이중 라벨
5. cross-doc / negative / vision 케이스 포함

## 8. 최종 권고

Jet-Rag 에 가장 어울리는 전략은 다음이다.

> **유료 모델은 눈이 필요한 곳과 질문을 쪼개는 곳에만 쓰고, 검색 자체는 무료 retrieval 엔진을 더 똑똑하게 만든다.**

이 방향이 제품 가치와도 맞다.

- 업로드 후 빠른 검색
- 비용 예측 가능성
- 복잡 질문 대응
- 출처 기반 신뢰성
- 향후 Local-first 전환 가능성

즉, "모든 것을 좋은 유료 LLM 에 맡기는 RAG"가 아니라, **로컬/무료 검색 엔진을 중심에 두고 유료 모델을 정밀 보강재로 쓰는 RAG**가 Jet-Rag 답다.
