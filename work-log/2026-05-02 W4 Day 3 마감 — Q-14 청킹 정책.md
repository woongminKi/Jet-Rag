### 2.2 핵심 변경 — `_SENTENCE_END` 재작성

#### Before
```python
_SENTENCE_END = re.compile(
    r"(?<=[.!?])\s+"                  # ← 일반 마침표 (false split risk)
    r"|(?<=[다요까죠습니다])[.!?]?\s+"  # ← char class 단일 char 매칭, 중복 다
    r"|\n\s*\n"
)
```

#### After
```python
_SENTENCE_END = re.compile(
    # (a) 한국어/괄호 + 문장부호 뒤 공백 — fixed 2-char lookbehind, `\s+` 만 매칭
    r"(?<=[가-힣\)\]][.!?])\s+"
    # (b) 문단 break
    r"|\n\s*\n"
)

# 4.2 보강 — 법령 인용 (`yyyy. m. d.`) placeholder 마스킹
_LEGAL_DATE_PATTERN = re.compile(r"(\d{2,4})\.\s+(\d{1,2})\.\s+(\d{1,2})\.")
```

**왜 단일 alternation?**
- `[가-힣]` (한국어 음절) 으로 모든 종결어미 (다·요·까·죠·습·니·네·군·지·…) 일반화 — 화이트리스트 불필요
- 직전 1 char 가 숫자/영문/약어이면 자연 차단 (lookbehind fail)
  - `Section 1. Intro` ✓ 차단 (`1` not Korean)
  - `vs. Acme` ✓ 차단 (`s` not Korean)
  - `2025. 7. 9.` ✓ 차단 + placeholder 추가 안전망
- 영문 sentence 분할은 의도적 미지원 (한국어 dominant 환경 trade-off, force-split 으로 _MAX_SIZE 보장)

**leftmost 매칭 버그 회피**:
- 이전 시도 `(?<=[가-힣\)\]])[.!?]\s+` 는 `[.!?]` 를 매칭에 포함시켜 leftmost (1 char 왼쪽) 에서 매칭됨 → 마침표 소실
- 수정: `[.!?]` 를 lookbehind 안으로 이동해 zero-width 로 처리 → 마침표 좌측 청크에 보존

### 2.3 `_apply_overlap` — 4.4 prefix overlap

```python
_OVERLAP_SIZE = 100  # 한국어 RAG 표준 50~150 중간

def _apply_overlap(pieces: list[str]) -> list[str]:
    if len(pieces) <= 1:
        return pieces
    out: list[str] = [pieces[0]]
    for i in range(1, len(pieces)):
        prev = pieces[i - 1]
        cur = pieces[i]
        prefix = prev[-_OVERLAP_SIZE:] if len(prev) > _OVERLAP_SIZE else prev
        # _MAX_SIZE 보장 — len(prefix) + 1(공백) + len(cur) <= _MAX_SIZE
        budget = _MAX_SIZE - len(cur) - 1
        if budget <= 0:
            out.append(cur)
            continue
        if len(prefix) > budget:
            prefix = prefix[-budget:]
        out.append(f"{prefix} {cur}".strip() if prefix else cur)
    return out
```

- 적용 위치: `_split_by_sentence` 의 마지막 (force-split 직후) — 같은 source section 안에서만
- merge 후 다른 section 끼리 overlap 안 함 (의미 섞임 risk 회피)
- `_to_chunk_records` 에서 idx > 0 청크에 `metadata["overlap_with_prev_chunk_idx"]` 기록

### 2.4 `_merge_short_sections` — 4.5 section_title swap

### 5.2 한계 + risk

1. **영문 sentence 분할 미지원**: `Hello world. Next.` 는 split 안 됨 → 긴 영문 doc 에서 force-split (_TARGET_SIZE 단위) 의존. 사용자 자산 분포 (한국어 dominant) 에서 risk 낮음.
2. **leftmost 매칭 버그 회피 후 회귀 risk**: 이전 잘못된 패턴은 마침표 소실 → 검색 시 청크 끝 마침표 누락. 수정 후 마침표 보존 → 기존 chunks 와 동등. 회귀 0 확인.
3. **법령 인용 패턴 좁음**: `2025. 7. 9.` 만 매칭. `2025. 7. 9.자` `2025년 7월 9일` 등 변형 미커버. 페르소나 A 의 법률 PDF (law sample3) 12 chunks 만 영향.
4. **overlap 메타 정확성**: `_to_chunk_records` 가 모든 idx > 0 에 `overlap_with_prev_chunk_idx` 표시. 실제로는 split 인접 조각만 overlap, merge 결과 인접은 overlap 없음 → 메타 misleading. P3 backlog (W4 trivial 처리 가능 시점에 정확화).
5. **dry-run worst-case (-66.8%) 의 의미**: 실제 재인제스트는 파서 출력 (다중 섹션) 에서 시작 → 실 변화는 더 작을 것. 하지만 정확한 측정은 본 적용 후 확인.
