### 4a. 018 데이터 이관 SQL

```sql
-- :owner = 본인 UUID (예: '7f3a9b12-4d56-7890-abcd-ef1234567890')
-- :legacy = '00000000-0000-0000-0000-000000000001'
BEGIN;

UPDATE documents
   SET user_id = '7f3a9b12-4d56-7890-abcd-ef1234567890'::uuid
 WHERE user_id = '00000000-0000-0000-0000-000000000001'::uuid;

UPDATE answer_feedback
   SET user_id = '7f3a9b12-4d56-7890-abcd-ef1234567890'::uuid
 WHERE user_id = '00000000-0000-0000-0000-000000000001'::uuid;

UPDATE answer_ragas_evals
   SET user_id = '7f3a9b12-4d56-7890-abcd-ef1234567890'::uuid
 WHERE user_id = '00000000-0000-0000-0000-000000000001'::uuid;

-- 검증
SELECT 'documents' AS tbl, COUNT(*) FROM documents WHERE user_id = '7f3a9b12-...'::uuid
UNION ALL
SELECT 'answer_feedback', COUNT(*) FROM answer_feedback WHERE user_id = '7f3a9b12-...'::uuid
UNION ALL
SELECT 'answer_ragas_evals', COUNT(*) FROM answer_ragas_evals WHERE user_id = '7f3a9b12-...'::uuid;

COMMIT;
```

## 트러블슈팅

### 1. JWT 가 비대칭(ECC) 인 경우
**증상**: 1a-4 에서 JWT Settings 에 "Legacy JWT Secret" 만 있고 "Asymmetric keys" 가 default 활성. 백엔드가 401 만 반환.

**대응**:
- 옵션 A — Supabase dashboard 에서 Legacy JWT secret 활성화로 임시 다운그레이드(이 경우 본 가이드 그대로 진행).
- 옵션 B — 백엔드 jwt_verify 를 JWKS 비대칭 모드로 전환. Railway ENV 에 `SUPABASE_JWT_ALGORITHM=RS256` (또는 ES256) + 추후 코드 보강. **재시도 sprint 필요** — 본 가이드 보류하고 알려주세요.

### 2. 본인 가입 후 redeem 실패 ("초대 코드가 유효하지 않습니다")
**원인 후보**:
- 017 미적용 → 503. SQL Editor 에서 `SELECT * FROM invite_codes` 가 에러 나면 017 재실행.
- 초대 코드 typo → 400. `SELECT code FROM invite_codes WHERE code='JETRAG-OWNER-001'` 확인.
- 이미 소진(used_by NOT NULL) → 409. 새 코드 INSERT 또는 `UPDATE invite_codes SET used_by=NULL, used_at=NULL WHERE code='JETRAG-OWNER-001'`.
- Bearer 토큰 추출 실패(JWT 비대칭 등) → 1번 트러블슈팅.

### 3. 가입 후 본인 inbox 가 비어 있음
**원인**: Phase 4a (018 SQL) 미실행. 데이터가 legacy UUID 에 있어 본인 계정에 안 보임. 4a 실행 → 새로고침.

### 4. 시크릿 창에서 `/login` 리다이렉트가 안 됨
**원인**: proxy.ts 가 Supabase ENV 미설정 시 graceful 통과. Vercel ENV 확인 + Redeploy 확실히 됐는지.

### 5. Phase 5 script 가 `move()` 호출에서 AttributeError
**대응**: 정상 — script 가 download/upload/delete 폴백 자동 진행. stdout 의 `moved=N` 카운트로 확인.

### 6. Realtime 채널이 SetAuth 후에도 0 event
**원인**: 020 PART 2 활성화 전이면 RLS publication 정책 부재 → 자연 deny. PART 2 적용 후 다시 시도.
