## 1순위 — search retrieval 단 정밀화 (cost 0, 1~2 day)

### 목적
G-A-204 의 ch 919 (요약표 데이터 part 2) 가 search top-10 밖 — vision adjacent boost (Batch 1 sprint 3) 가 효과 0 인 이유 (candidates pool 자체 부재) 후속 sprint.

### 핵심 발견 (직전 sprint)
- Batch 1 sprint 3 의 vision adjacent boost: candidates pool 안에서만 score propagate → 919 가 retrieval 단계 (`/rpc/search_candidates`) 에서 제외되면 boost 발동 불가
- 즉 retrieval 자체 (BM25 + dense embedding) 가 919 를 top-100+ 밖으로 보냄

### 가능 접근

| 옵션 | 작업 | 효과 | risk |
|---|---|---|---|
| A | **chunk text augmentation** — search 시 인접 chunk text prepend (BM25/dense 입력 보강) | 919 의 sparse text 가 918 의 caption 으로 보강 → matching ↑ | search latency ↑ (per-query chunk fetch) |
| B | **ingest 시 merge** — 같은 (vision page) 의 chunks 통합 | 근본 해결, 통합된 chunk 가 retrieval 단 진입 | reingest 필요 (cost + 데이터 변경) |
| C | **dense embedding 가중치** — 짧고 numeric chunks 의 dense vector 를 caption 가중치로 보강 | 정확 매칭 ↑ | embedding 재생성 필요 |
| D | **다른 검색 path 추가** — vision-derived chunks 만 별도 fallback retrieval | 회귀 risk 작음 | 인프라 복잡도 ↑ |

### 권고: Option A (chunk text augmentation)
- 작업량 1 day, cost 0
- 운영 코드 변경 + ablation 측정
- ch 919 회복 검증

### 진입 조건
- 사용자 명시 (운영 코드 변경)
- BGE-M3 HF API 안정 (한계 #3/#8/#12)

### 산출물
- `api/app/services/search_augment.py` 신규 또는 search.py 통합
- ablation 측정 (D4 도구 재실행)
- work-log
