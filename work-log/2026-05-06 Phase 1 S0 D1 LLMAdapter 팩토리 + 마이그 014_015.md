### 13.1 P1-1 — extract.py module-level vision factory 호출 lazy 화

**현상**: D1 ship 의 `_image_parser = ImageParser(captioner=get_vision_captioner("image_parse"))` 가 module-level 즉시 실행 → `JETRAG_LLM_PROVIDER=invalid` 또는 `JETRAG_LLM_PROVIDER=openai` (key 부재) 시 module import 자체 실패 → API 서버 startup fail.

**fix**:
- `_image_parser` / `_pptx_parser` / `_PARSERS_BY_DOC_TYPE` 모두 `@lru_cache(maxsize=1)` lazy 함수 (`_get_image_parser` / `_get_pptx_parser` / `_get_parsers_by_doc_type`) 로 전환
- 가벼운 파서 (PyMuPDF/HWPX/URL/HWP/HWPML/DOCX) 는 module-level 인스턴스 유지 — 외부 호출/네트워크 0 이라 안전
- `run_extract_stage` / `_reroute_pdf_to_image` / `_enrich_pdf_with_vision` 호출처는 `_get_image_parser()` / `_get_parsers_by_doc_type()` 로 대체

**factory.py 변경 X (의도)**: `_resolve_provider` 가 invalid provider 까지 fallback 하면 `test_unknown_provider_raises_in_get_*` 회귀 발생 — 호출 시점의 명확한 차단을 보존하는 게 사용자 경험에 유익. import-time 안전성은 호출처 lazy 화로 충분히 확보.

### 13.2 P1-3 — gemini_vision._parse_usage_metadata 의 image_tokens 활성화

**현상**: Gemini SDK 가 `usage_metadata.prompt_tokens_details: list[ModalityTokenCount]` 로 IMAGE/TEXT modality 분리 제공함에도 D1 구현은 `image_tokens=None` 하드코딩 → 마이그 014 의 image_tokens 컬럼 항상 NULL.

**SDK 실측 결과** (`google.genai.types`):
- `GenerateContentResponseUsageMetadata.prompt_tokens_details: Optional[list[ModalityTokenCount]]` 실존
- `ModalityTokenCount.modality: Optional[MediaModality]`, `token_count: Optional[int]`
- `MediaModality` enum: `TEXT` / `IMAGE` / `VIDEO` / `AUDIO` / `DOCUMENT` / `MODALITY_UNSPECIFIED`

**fix**: `_extract_image_tokens(metadata)` 헬퍼 추가. `prompt_tokens_details` 를 순회하며 IMAGE modality 의 `token_count` 만 합산. enum object (실 SDK) 와 str ("IMAGE", 모킹) 양쪽 graceful 처리. 합 0 또는 details 빈 list 시 `None` (NULL 컬럼).

**컬럼 의미 결정**: `prompt_tokens` = SDK 의 `prompt_token_count` (텍스트+이미지 합산) 그대로 보존. `image_tokens` 는 정보용 별도 컬럼. 단가 계산은 `prompt_tokens` 단일 단가 적용 (회귀 0). 이 결정으로 `_estimate_cost` 시그니처 보존.

### 14.4 HyDE cache key 보강

**이슈**: 이전 `_cache: OrderedDict[str, str]` (key = query only) → 모델 변경 시 같은 query 가 stale 결과 반환. D2-D 의 모델 swap 시 회귀 위험.

**fix**: key = `(model_id, query)` tuple. `_model` 속성 → 클래스명 fallback 으로 robust. 회귀 보호 단위 테스트 3건 (test_hyde_cache.py):
- 다른 model_id 시 cache 격리 (LLM 호출 각각 1회)
- 같은 model_id + 같은 query 시 LLM 호출 1회만
- `_model` 속성 부재 provider 도 classname fallback 으로 cache 동작
